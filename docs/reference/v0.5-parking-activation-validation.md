# v0.5 Parking 激活验证

本文记录 Issue #109 在 `main@db3b86c2db19ec89f3d7478bbd601611d124bbb4` 上的功能、原子性、分配与性能证据。它只覆盖 ParkingStop、arrival、parking-aware traversal、route-completion release 和 capability activation；完整 v0.5 milestone 验收已由 #110 固化在 [`v0.5-parking-validation.md`](v0.5-parking-validation.md)，本文不替代该组合证据。

## 实现边界

- ParkingStop 与 SignalStop、RouteEnd 使用同一 spatial candidate reducer；严格度按 travel、speed 比较，完全相同时按 Signal -> Parking -> RouteEnd 归因。
- `reserved_count == 0` 在纵向构建与车辆推进外层选择编译期专化快速路径，不进入逐车 Parking lookup。
- nonzero Reserved 通过 reverse binding 和 cached first-reachable target 在既有 stable vehicle loop 中读取；约束以稳定顺序保存为 sparse `(vehicle, ParkingStopConstraint)`，推进时用单调游标合并，不新增 full-S/full-V Parking pass。
- arrival 与 route-completion release 先写 candidate event/delta；所有车辆和 signal candidate 成功、全部 touched pair local invariant 通过后才提交 Parking authority。
- legacy `ParkingVehicleCapabilityUnavailable` variant 保留兼容，但合法 production world 不再可达。

## 功能与原子性证据

主要命令：

```text
cargo +1.96.0 test -p laneflow-core --lib --all-features
cargo +1.96.0 test -p laneflow-core --test parking_activation --all-features
cargo +1.96.0 test -p laneflow-core --test parking_runtime --all-features
```

覆盖内容：

- far/near/arrived waiting、Dormant、repeated route occurrence 和 epsilon boundary；
- Signal-before-Parking、Parking-before-Signal、exact tie、spatial + Following 双 projection；
- multi-edge final travel、entry boundary snap、arrival one-shot、explicit commit 后 Parked exclusion；
- route completion 的 release-before-completed 总序；
- vehicle advance 后 failure injection、全 world 原子不变以及 retry replay；
- repeated route 的 first-reachable target 对照独立 route-scan oracle；统一 spatial reducer 对照三个 provider 单独计算结果的 128-case property oracle；
- command-created Arrived 不补发 step arrival event；legacy guard 对合法 world 不可达。

## 分配证据

命令：

```text
cargo +1.96.0 test -p laneflow-core --test parking_runtime_allocations --all-features --locked -- --ignored --test-threads=1
```

四项测试通过：

- warm Reserved approach 的无事件 steady tick：0 allocation / 0 reallocation；
- warm Arrived waiting 与 occupied-only steady tick：0 allocation / 0 reallocation；
- Parking snapshot 与 warm command：0 allocation / 0 reallocation；
- 确定性事件 tick replay：除返回的 event buffer 一次 allocation 外无其他 allocation，且无 reallocation。

## 10k 静态回归

Criterion workload 为 10k vehicles、60 ticks、16 ms，20 samples，1 s warm-up、5 s measurement。每次切换 main/candidate 都先执行 `cargo clean -p laneflow-core --release` 再重建，避免不同 worktree 复用同名 benchmark binary。以下是三组低噪声相邻配对的 point estimate：

| 配对     | main empty (ms) | candidate empty (ms) |      delta | main all-Vacant (ms) | candidate all-Vacant (ms) |      delta |
| -------- | --------------: | -------------------: | ---------: | -------------------: | ------------------------: | ---------: |
| 1        |          59.770 |               58.816 |     -1.60% |               60.939 |                    59.411 |     -2.51% |
| 2        |          60.755 |               59.255 |     -2.47% |               61.868 |                    61.341 |     -0.85% |
| 3        |          60.810 |               59.240 |     -2.58% |               60.419 |                    61.964 |     +2.56% |
| 三轮中位 |               - |                    - | **-2.47%** |                    - |                         - | **-0.85%** |

empty 与 all-Vacant 均低于 5% 目标线。一次 candidate 样本出现 76.98–91.62 ms 的异常宽置信区间，且相邻 main 同时升高；该外部负载污染样本保留在 Criterion artifact 中，但不纳入 matched low-noise median。更早一次共享 target、未强制重建导致复用错误 worktree binary 的试运行同样明确作废。

## 100k 缩放

正确性命令：

```text
cargo +1.96.0 test -p laneflow-core --test parking_runtime_scale hundred_thousand_vehicle_parking_runtime_smoke_preserves_counts --all-features -- --ignored --exact
```

结果为 1 passed。100k / 100% Reserved Criterion point estimate 为 899.45 ms/60 ticks；相对 10k / 100% Reserved 三轮中位 76.38 ms，缩放为 **11.78x**，低于 20x 阻断线。raw baseline 名为 `parking109-final-activation-100k`。

## 环境说明

- CPU：AMD Ryzen 9 9955HX，16 cores / 32 logical processors；Windows 本机执行。
- 静态配对第 2/3 组及 activation/100k benchmark 使用 High process priority；所有判定使用 point estimate，置信区间与 outlier 信息保留在 Criterion artifact。
- Rust 构建多次报告 Windows incremental session directory `Access denied (os error 5)`，但命令均成功完成；该提示仅使下一次构建不能复用对应 incremental cache，不影响测试或 benchmark binary。
