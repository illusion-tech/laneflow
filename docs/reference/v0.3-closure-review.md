## 3. 设计与架构契约

- ADR 0003 保持 fixed tick、同版本同环境确定性和失败原子性边界。
- ADR 0005 的 opaque typed handle、generation、resolver 和 stable update order 继续成立；M0.3 没有把 external ID 带回 tick hot path。
- ADR 0006 固化 IIDM comfort controller、独立 hard safety envelope、ballistic integration、最终 geometry projection 和不公开 controller trait 的边界。
- ADR 0007 固化 `laneflow-data -> laneflow-core` 依赖方向、private wire DTO、Core normalization 和无文件系统 loader 边界。
- ADR 0008 替代早期双版本方案：1.0 前只维护一个 active data format，当前为 `0.3`；旧版只被明确拒绝，不保留 production compatibility path。
- Accepted `vehicle-following.md` 与 `data-loading.md` 对 profile、状态、版本闸口、loader、错误、测试和性能责任描述一致。

Core 不以 wire format 版本建模。`laneflow-core` 只拥有当前 domain types 和 runtime invariant；`laneflow-data` 只公开单一当前 `LoadedPackage`。M0.2 建立的 handle、lane graph 与 route 领域语义成为当前 Core 的组成部分，不表示主线同时维护 v0.2/v0.3 两套实现。

## 4. 实现与不变量审阅

- Vehicle Profile registry 不可变，external ID 只在边界解析为 `VehicleProfileHandle`。
- VehicleState 使用 front-bumper progress、finite signed acceleration 和明确的 Active/Stopped/Completed 状态语义。
- Occupancy 使用可复用扁平 scratch，按 physical edge 和稳定 update order 构建；leader detection 沿车辆已选 route 解析。
- Tick 基于 tick-start snapshot 计算 candidate state，依次应用 IIDM、safe-speed、ballistic integration 和确定性 no-overlap projection，成功后原子提交。
- Candidate state、occupancy、leader 和 longitudinal scratch 均跨 tick 复用；成功和失败路径都会恢复可复用容量，semantic equality 不比较派生 scratch。
- Spawn/despawn、route register/remove、stale handle、slot generation、route transition/completion 和事件顺序均有定向测试。
- 属性测试覆盖线性 platoon 的输入顺序不变性、finite/non-negative state、no-overlap 和逐 tick 精确确定性；复杂拓扑与生命周期由可解释的定向测试覆盖。

源码审阅未发现 `unsafe`、`TODO`、`FIXME`、`HACK`、`todo!()` 或 `unimplemented!()` 遗留。该搜索只证明当前受检源码没有这些标记，不替代行为测试或安全审计。

## 5. 数据格式与 Schema

当前 active tree 只有：

- `schemas/laneflow-data-v0.3.schema.json`；
- `examples/data/v0.3-profile-baseline.laneflow.json`；
- 一套 private v0.3 wire DTO 和单一 `LoadedPackage`；
- 对旧版、未来版和缺失版本的结构化拒绝测试。

生产 loader 先解析最小 version header，只允许精确的 `formatVersion: "0.3"` 进入当前 shape 与 Core normalization。v0.2 只作为 stale-version 负向输入出现，不存在 v0.2 DTO、schema、fixture、normalization 分支或 public version enum。

## 8. 发现项处置与剩余边界

- Tick candidate state 曾每步重新分配：由 #86 修复并完成成功/失败路径容量复用测试与三轮性能复核。
- Core README、架构和设计追踪存在“历史里程碑”与“当前实现”混写：随本收口报告同步修正，不再把 v0.2 描述成并行 active 实现。
- 百万级城市运行时需要 partition、parallel、multi-rate 和 mesoscopic/aggregate 架构：由 [#72](https://github.com/illusion-tech/laneflow/issues/72) 研究，不阻断 M0.3。
- 源代码许可与依赖审计由 #56 跟踪，GitHub 代码/凭据扫描由 #88 跟踪；二者是发布治理责任，不能因 M0.3 收口而丢失。
- Windows 偶发 incremental cache `os error 5` 只影响后续缓存复用；在命令退出码为 0 且测试/clippy/doc 完成时，不改变验证结论。

## 9. 影响结论

- **Core API**：M0.3 有意以 profile handle、`current_speed` 和 `applied_acceleration` 替代 pre-1.0 临时 vehicle API；M0.2 的 typed handle、route lifecycle 和 determinism 契约保持成立。
- **Data format**：依据 ADR 0008 直接以当前 v0.3 替换 v0.2，不提供 runtime compatibility shim；这是 pre-1.0 明确决策，不是遗漏兼容实现。
- **Adapter API**：未新增或冻结 Adapter ABI。未来 Adapter 可消费 Core handle/state/event，并按需通过 `laneflow-data` 加载当前 package，但不得复制交通规则。
- **示例/演示**：当前 v0.3 fixture 覆盖 loader/schema/domain 闭环；M0.3 不交付引擎内可视化示例。

## 10. 结论

截至本次审阅，未发现阻断 M0.3 Vehicle Following 收口的正确性、确定性、性能、数据契约、架构或治理缺口。#17 的所有实施 sub-issues 已完成；收口 PR 通过审阅、CI 和 G3 后，可以合并并执行 #17 G4、Project Done 与 milestone 关闭。

后续版本可以把本基线作为稳定输入，但不得把它解释为尚未实现的 lane changing、signals、intersection、parking、Adapter 或城市级调度能力已获批准。
