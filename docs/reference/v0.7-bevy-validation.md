## 1. 结论

#171 在不改变 Core/Data/Spatial/Adapter public API、不改变数据格式、也不增加 production dependency 的前提下，为 `laneflow-bevy` 建立了 production validation 路径：

- 仓库现有 campus manifest、traffic artifact 和 spatial artifact 通过 production loader 的 size/digest 校验，完成 load → Core fixed step → Spatial batch → Bevy local/global Transform E2E；
- 相同 `32 ms` 总时间按 `[7, 9, 16] ms` 与 `[32] ms` 分块后，Core committed state、稳定车辆/pose 顺序和最终 Transform 完全一致；
- 预热且容量稳定的完整 Bevy `PostUpdate` 在 10k/100k 下均为零 allocation/reallocation；
- source commit `d7e8b1e2313fdadb318433acaa6ed4c998683456` 的正式 Rust 1.96 固定机结果为 10k `3.067 ms`、100k `35.852 ms`、扩展 `11.691x`，通过 `4/40 ms` 与 `12x` Gate；
- 共享 CI 执行 correctness/determinism、allocation 和 benchmark compile，但不把 noisy wall-clock 当作共享 runner 或跨平台 SLA。

完整逐轮机器数据保存在 `v0.7-bevy-performance-evidence.json`。

## 2. 被测边界

每个性能样本直接运行一次已经预热的完整 Bevy `PostUpdate` schedule：

```text
Spatial pose batch extraction
  + Adapter frame/mapping/entity/parent/Transform validation
  + Bevy local Transform writes
  + TransformSystems::Propagate
```

不计入 fixture/Entity 构造、Core fixed tick、交通求解、`LaneFlowOuterFrame`、其它 schedule、renderer、window、audio、asset、scene 或 Gizmos。每档夹具包含一条合法 canonical 直线、与档位相同数量的唯一 live Core vehicle/Entity、一个单位缩放 frame-root，以及预留到目标容量的全部 Session-owned buffer。

scale fixture 使用 `0.1 m` 车辆长度和 `0.25 m` 间距，使 100k 个唯一 vehicle 能合法装载在 `25,000 m` 直线上。该尺寸只控制测试装载，不代表真实交通构成；被测 Adapter 每记录工作量、映射与 ECS 层级仍与 production 路径一致。

## 3. Campus E2E 与确定性

E2E 直接编译并加载以下仓库制品，不复制测试专用 JSON：

- `examples/data/v0.1-campus.scenario.json`；
- `examples/data/v0.5-empty-signals-and-parking.laneflow.json`；
- `examples/data/v0.1-campus.spatial.json`。

`laneflow_data::from_scenario_json_slice` 先校验 manifest 声明的 artifact ref、size 与 SHA-256，再生成 `InitialTrafficData` 和按 Core edge 稳定顺序排列的 Spatial 输入。测试创建 main/loop 两辆 active vehicle、frame-root 和直接 child proxy，随后比较两种 outer-frame 分块的：

- `CoreWorld`、tick `2`、time `32 ms` 与零 backlog；
- 按 Core committed vehicle order 解析的 Entity 顺序；
- local translation/rotation 与 propagated global translation；
- pose/mapped/unbound/applied presentation report。

0/1/多 fixed step、catch-up backlog、bind/unbind/rebind、partial mapping、Y-up / `-Z` forward、rebase/token mismatch、stable first-error 与失败零写入继续由现有 `fixed_schedule`、`presentation` 和模块测试承担。#171 在同一 package/workspace test run 中执行它们，不另造第二套生产逻辑。

## 4. 零分配 Gate

`stats_alloc` 测试在 fixture setup 和一次额外预热后，以串行 ignored test 只测一个完整 `PostUpdate`。两档均得到：

| 车辆/Entity 数 | allocation | reallocation | allocated bytes | reallocated bytes | 结论 |
| -------------: | ---------: | -----------: | --------------: | ----------------: | ---- |
|         10,000 |          0 |            0 |               0 |                 0 | Pass |
|        100,000 |          0 |            0 |               0 |                 0 | Pass |

复现命令：

```powershell
cargo +1.96.0 test -p laneflow-bevy --test presentation_allocations --locked -- --ignored --test-threads=1 --nocapture
```

## 5. 固定机性能 Gate

正式环境为 AMD Ryzen 9 9955HX（16C/32T）、64 GiB、Windows `10.0.29617`、`x86_64-pc-windows-msvc`、平衡电源方案、Rust/Cargo `1.96.0`、LLVM `22.1.2`。运行前未发现 UnrealBuildTool、ShaderCompileWorker、MSBuild、`cl` 或 `link` 后台编译进程。

每档预热 8 次，执行 5 轮、每轮 80 个样本；结果取 5 个 per-round p95 的中位数。正式运行是 source commit 上第一次完成后台审计的 Rust 1.96 运行；此前 Rust 1.97 smoke 只证明入口可执行，不参与证据选择。

```powershell
$env:LANEFLOW_BEVY_PERF_GATE = '1'
cargo +1.96.0 test --release -p laneflow-bevy --test presentation_performance --locked -- --ignored --test-threads=1 --nocapture
```

| 车辆/Entity 数 | 五轮 p95 中位数 | 绝对 Gate |    余量 | 结论 |
| -------------: | --------------: | --------: | ------: | ---- |
|         10,000 |      `3.067 ms` |    `4 ms` | 23.333% | Pass |
|        100,000 |     `35.852 ms` |   `40 ms` | 10.369% | Pass |

10k→100k 的 p95 扩展为 `11.690840x`，低于 `12x` Gate。它接近但没有越过冻结边界，因此 evidence 保留全部五轮原始 p95/median，不放宽阈值，也不选择较快的 Rust 1.97 smoke 结果。

Criterion 入口 `crates/laneflow-bevy/benches/presentation.rs` 复用同一 fixture，供本地诊断。共享 CI 只执行 `cargo bench -p laneflow-bevy --benches --no-run --locked`，不把 Criterion estimate 当作固定机通过证据。

## 6. CI、依赖与适用边界

共享 Rust CI 在既有 workspace/MSRV/fmt 流程之外新增：

- 串行显式执行 Bevy Adapter ignored allocation test；
- 编译 Bevy Adapter Criterion benchmark；
- campus E2E 与确定性测试随 `cargo test --workspace --locked` 执行；
- 既有 cargo-deny job 继续以 `--all-features` 审计 advisories、bans、licenses 与 sources。

新增依赖全部是 `laneflow-bevy` dev-only：`laneflow-data`、`stats_alloc` 和 `criterion`。production/default modular Bevy graph 仍只有 `bevy_app`、`bevy_ecs`、`bevy_time`、`bevy_transform`，没有 renderer/window/audio/asset/scene/Gizmos。

绝对毫秒只对本文机器、工具链、source commit、release profile、fixture 和测量边界成立。它不是 WASM、其它引擎、其它操作系统或跨平台 SLA，也不代表包含 Core 交通求解和 renderer 的整帧预算。
