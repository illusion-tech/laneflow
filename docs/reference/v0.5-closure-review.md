## 3. 设计与架构契约

- ADR 0010 固化 Parking binding、vehicle lifecycle/position authority 与 Core/Adapter 分层；`ParkingSpace` 是占用事实源，`ParkingArea` 不是独立运营或容量 authority。
- 停车场泊位与 standalone 专用路边泊位使用相同 individual-space lifecycle；lot/curbside kind 不进入 Core 行为分支。
- caller 负责选位、route 与明确 occurrence；Core 不自动选位、寻路、排队或调度。
- static Parking 通过 current data 0.5 进入 immutable registry；reservation、occupancy、runtime handles 和 initial parked vehicles 不持久化回 external package。
- `VehicleStatus::Parked` 保留 vehicle identity，但 parking pose 成为位置 authority；Parked vehicle 不参与 lane occupancy、leader、motion 或 traversal。
- ParkingStop 与 SignalStop、RouteEnd、following、safe-speed、no-overlap 共享 fixed-tick candidate pipeline；相同 spatial boundary 的 attribution order 为 Signal、Parking、RouteEnd。
- successful tick/command 只提交完整 candidate；失败、no-op、invalid handle/reference 和 failure injection 不留下 partial Parking、vehicle、route reference 或 event state。
- Adapter 仍只消费 Core state/query/events 做表现；v0.5 没有新增或冻结 Parking Adapter ABI。

Accepted `parking-system.md`、ADR 0010、data format/loading、route、Signals、Vehicle Following、architecture 与当前代码在这些边界上一致。

## 4. 实现与不变量审阅

- `ParkingRegistry` 使用 opaque dense handles、稳定 input/member order 和 immutable resolver；area/space identity、anchors、geometry 与 foreign-graph rebind 在 initialization 完成。
- `ParkingRuntimeState` 保存 dense space/binding/count authority；snapshot 是 borrowed committed view，global/area counts 增量维护。
- reserve、cancel、commit、leave、rebind 与 parked-spawn 六类同步 command 保持 fixed error priority、caller order、atomicity 与 deterministic replay。
- Reserved Active vehicle 选择 cursor 后 first-reachable entry occurrence；Arrived one-shot、explicit commit、route-completion release、despawn cleanup 与 leave/resume 都有定向和 property/oracle 覆盖。
- command spatial index 只缩小候选，不拥有最终 authority。pathological fastest-vehicle removal 使用按需 exact speed heap，把 fixed batch 最坏扩展从 `O(B x V)` 收敛为 `O(V + B log V)`；正常 successful tick 仍只维护 scalar max。
- runtime 不扫描全部 Parking spaces/signals，不保存 per-vehicle ParkingSpace 副本；10k all-Vacant registry 不进入 step hot path。
- `parking_lifecycle` 是 engine-independent loader-to-Core example：production loader -> `InitialTrafficData` -> `CoreWorld` -> reserve/arrival/commit/leave/resume，不是绕过 data layer 的 Core-only fixture。

对 `crates/**/*.rs` 的精确搜索未发现 `unsafe` keyword、`TODO`、`FIXME`、`HACK`、`todo!()` 或 `unimplemented!()` 遗留。该搜索只证明受检 Rust 源码没有这些标记，不替代行为、性能或安全测试。

- Windows WPR 是含 setup/warm-up/analysis 的 process-wide corroboration；operation counter、matched scaling 与 profiler 必须组合解释，不把单次 ETL 百分比当作独立门禁。

## 9. 影响与迁移结论

- **Core API**：v0.5 有意新增 Parking handles/registry/snapshot、commands/records/errors/events、`VehicleStatus::Parked` 与 Parking-aware lifecycle；fixed tick、typed handle、route occurrence、Signals authority、Following safety 与失败原子性继续成立。
- **Data format**：依据 ADR 0008，current format 从 0.4 直接切换到 0.5，新增必填 closed `parking { areas, spaces }`；不提供 production compatibility shim，旧 current 只被明确拒绝。
- **Adapter API**：未新增或冻结 Parking Adapter ABI。未来 Adapter 可消费 authoritative query/events 和 parked pose，但不得复制 reservation、occupancy、route、permission 或 safety authority。
- **示例/演示**：两个 canonical fixtures 与 `parking_lifecycle` 覆盖 schema、production loader、Core normalization 和完整生命周期；不包含引擎渲染、free-space maneuver 或 authoring UI。
