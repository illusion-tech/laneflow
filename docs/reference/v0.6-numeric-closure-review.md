## 1. 目的与结论边界

本文保存 #122 数值切片完成独立收口时可长期复核的事实。GitHub Issue / PR 继续负责 Gate、Project、审阅线程、CI 和合并时序等动态证据；本文不替代 #128、#122 或 #120 的 G3/G4 记录。

本次审阅证明数值切片已经形成闭环：ADR 0014 接受目标 `f32` 分层和迁移/回退门槛，#125–#127、#140/#141 建立领域所有权、产品范围、误差、路线布局、内存与性能证据，#144 在完整生产布局中执行候选迁移并因主性能门槛失败而完整回退。当前生产 Core/Data 的正式结论是继续使用 current-f64 与 Traffic Data v0.5，而不是等待自动恢复的半迁移状态。

这只是 v0.6 的数值切片收口，不表示 Spatial 或整个 v0.6 已完成。Spatial 继续按 ADR 0015 使用每轴 `±16_384 m` 的有界 canonical（标准）`f32` frame（坐标框架），其实施与独立收口由 #123、#133–#138 负责。v0.7 Adapter、Bevy 集成和 #72 的城市级分区、并行、多频率与中观运行时均不在本文完成边界内。

## 3. 当前生产事实

### 3.1 Core 与领域阈值

当前主线公共数值权威仍为：

- `EdgeLength(f64)`；
- `EdgeProgress(f64)`；
- `Speed(f64)`；
- `Acceleration(f64)`；
- Vehicle Profile、Parking、占用、纵向和命令空间中的对应 current-f64 值。

#125 已删除含义宽泛的两个旧公开常量，并把生产判定拆成九个 crate-private（仅 crate 内可见）所有者。九个当前值均保持 `1.0e-9`，现有比较方向、精确零状态、边界、状态、事件与失败原子性不变。生产代码不使用通用近似比较辅助函数、动态相对容差或运行时末位单位（ULP）阈值。

### 3.2 Data、schema 与候选清理

Traffic Data 的唯一有效版本仍是 `formatVersion: "0.5"`：

- `CURRENT_FORMAT_VERSION` 为 `"0.5"`；
- 活动树只有 `schemas/laneflow-data-v0.5.schema.json` 和 v0.5 Traffic 固定样例；
- 没有有效的 Traffic Data v0.6 loader、schema、固定样例或版本分派；
- 没有生产精度开关、同字段双权威、弃用别名、自动迁移或值域外 `f64` 回退；
- #144 的候选实现和回退链由 Git 提交及远端只读证据分支保存，不构成生产入口。

Spatial/Scenario v0.1 是独立数据 family（数据族），不等于 Traffic Data v0.6，也不改变上述 Traffic 当前版本结论。

### 3.3 Spatial 的独立精度边界

ADR 0015 已取代旧 Spatial `f64` 精度条款。当前 Spatial 公共 primitive 是 `CanonicalPoint3F32`、`CanonicalVector3F32` 和 `CanonicalUnitVector3F32`；canonical 点每轴范围为 `[-16_384 m, 16_384 m]`。

Core/Data 的 no-go 不回退 Spatial：current-f64 Core 进度可以作为有效输入被 Spatial 消费，但不能把 Spatial 顶点、弧长或位姿提升为第二套 `f64` 权威。反过来，Spatial 或 Adapter 的表现值也不能反馈写入 Core 状态。

## 4. 已接受目标与当前实现的分离

| 数值职责                    | ADR 0014/0015 接受的目标                                   | 当前生产                     | #144 结果                               |
| --------------------------- | ---------------------------------------------------------- | ---------------------------- | --------------------------------------- |
| 单 edge 长度与单值热状态    | 有产品范围的受检 `f32`                                     | current-f64                  | 候选回退                                |
| edge 进度                   | 比较补偿残差感知 `f32` 与单个 `f64` 进度，最终只留一个权威 | 单个 `f64`                   | 候选 B 胜出候选 A，但整体门槛失败后回退 |
| route 派生距离              | 规范化边长派生的分段局部 `f64` 前缀                        | current-f64 路线索引         | 目标布局随候选回退                      |
| Traffic Data                | 原子切换为唯一 v0.6                                        | 唯一 v0.5                    | v0.6 候选完整移除                       |
| 九领域目标阈值              | 固定、带单位、领域专用绝对值                               | 九个 `1.0e-9` current-f64 值 | 目标值未进入生产                        |
| Spatial canonical 几何/位姿 | 有界 `f32` frame                                           | 有界 `f32` frame             | 不受 Core/Data no-go 影响               |

ADR 0014 保持 `Accepted`，因为它接受的是目标设计、验证门槛与失败后的回退规则；#144 的 no-go 正是该规则的有效执行结果。ADR 0012 保持 `Superseded`：它继续描述迁移前及回退后的 current-f64 生产状态，但不再是未来目标权威的决策来源。

未来若重启 Core/Data 数值迁移，必须新建 Issue、重新进入 G1，并提供相对 #144 足以跨过同等或更严格收益门槛的新实现或硬件证据。不得直接恢复 #144 的实验提交、降低正确性预算、保留两套生产权威，或把 current-f64 自动解释为永久目标。

## 5. 正确性、内存与性能裁决

### 5.1 被修正的研究前提

#140 证明旧补偿候选只在写入进度时保存残差，却在前车间距、edge 剩余距离、边界和快照中只读取高位分量。补全 `effective = f64(high) - f64(residual)` 的所有读取路径后，候选通过既有连续误差与离散行为预算；因此误差根因不是“敏感控制必然需要 `f64`”，而是不完整的残差读取语义。

10 km 单 edge 防御性上界也不是为了用一个 edge 表示整条长道路。edge 表达拓扑、规则、控制与几何连续的有向车道段；更长道路应由 route 组合多个 edge。#140 的 1 万/10 万车辆矩阵没有发现拆 edge 必然带来稳态、构造或受控跨界性能损失。

### 5.2 #127 冻结输入

#127 在线下使用相对误差和 ULP 标定目标 `f32`，并产出固定、带单位、领域化的候选阈值；这些研究工具不进入生产 predicate（判定函数）。路线候选比较选择分段局部 `f64` 前缀：分块局部 `f32` 查询在 5/6 轮稳定变慢，配对中位数回退 `26.06%`，同时对路线密集完整布局只额外节省约 `0.92%`–`0.98%`，不足以支付复杂度。

### 5.3 #144 生产裁决

#144 使用完整生产布局、相同正确性矩阵和六场景 × 两规模 × 三生命周期阶段的 36 项常驻内存账本比较候选：

- 候选 A（补偿残差感知进度）在优化前五轮相对 current-f64 慢 `9.310%`；应用相同热路径优化后仍比候选 B 慢 `8.321%`，且没有额外完整内存收益，予以淘汰；
- 候选 B（`f32` 计算与单个 `f64 EdgeProgress`）通过正确性和内存护栏；36 项账本最差增加 `1.700%`，低于 `5%` 退化上限，也没有增加超过 `2%` 的未解释记录；
- 候选 B 的 14 项稳态配对中位比值等权几何平均为 `0.9574286156`，即耗时降低 `4.257138%`，低于冻结的 `5%` 主门槛；
- 主门槛已经失败，构造、命令受检转换和 route-distance 查询的次级矩阵不能推翻 no-go，因此按预先规则停止执行。

最终结论是 `rollback-current-f64`，不是“接近门槛所以酌情通过”。单机结果也不是跨平台服务等级承诺（SLA）；它只裁决本次实现、固定机器和冻结协议下是否值得承担跨 Core/Data 的破坏性迁移。

## 8. 影响与剩余边界

- **Core API**：无新变化；当前仍为 current-f64。#125 删除两个旧公开常量是已完成的既有破坏性变更，不由本收口重新实施。
- **Data format**：无变化；Traffic Data v0.5 继续是唯一有效版本。Spatial/Scenario v0.1 是独立数据族。
- **Spatial API**：无变化；继续采用 ADR 0015 的有界 canonical `f32`。
- **Adapter API**：无新契约；后续 Adapter 只消费有效进度和 canonical 位姿，不观察残差分量，也不反写 Core/Spatial。
- **依赖与许可证**：本收口不新增或更新依赖；`half` 继续只作为既有研究/测试开发依赖，不承担生产算术。
- **已知限制**：研究模型不等于生产迁移；单机基准不等于跨平台 SLA；current-f64 不自动成为永久目标；#72 不属于 v0.6；Spatial 和 v0.6 整体仍需独立完成。

## 9. 结论

截至本次审阅，没有发现阻断数值切片收口的正确性、确定性、数据契约、性能裁决、架构或治理缺口。关键结果不是“`f32` 永远不适合 LaneFlow”，而是本次完整生产候选没有达到预先冻结的实质收益门槛，因此可靠地回退 current-f64。

#128 Delivery PR 通过 Copilot、人工最终差异审阅、CI、CodeQL、Dependency policy（依赖策略）、审阅线程与 G3 后，可以按顺序执行 #128 G4，再解除 #122 阻塞并完成 #122 G4。#120 只能汇总数值切片完成，不能据此关闭 v0.6 或提前启动被 v0.6 整体阻塞的 v0.7。
