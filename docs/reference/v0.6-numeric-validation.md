## 1. 证据状态

本文是 #122 G1、#140 后续复核与 #141 目标决策的可复现证据记录。#122 已完成生产数值面盘点、产品/误差范围、旧版/保持局部性的 `f64` 基线、研究专用 `f32`/混合候选、1 万/10 万车辆运行时与内存、贴近生产的 `f64` 参照、边界/确定性/长时累计、`f16`/整数定点矩阵和跨层迁移判断。

2026-07-18 的 #140 进一步证明：#122 的补偿 `f32` 候选只在累加时维护补偿残差，却没有在前车间距、edge 剩余距离、边界和快照中使用它；因此“只有 `f64` 进度的混合候选通过”和“`f32` 没有性能收益”不再是当前有效结论。新的补偿残差感知 `f32` 在本研究模型的 1 万/10 万车辆矩阵通过现有严格预算，并在同轮 Criterion 中快于 `f64`。

#141 随后接受 ADR 0014 的目标契约，但没有把研究结果伪装成已完成的生产迁移。当前生产 Core/Data v0.5 继续保持 ADR 0012 的 `f64` 实现状态，直到原子转换、完整生产判定基准、扩展常驻内存账本、性能闸口与 Data/API 迁移全部通过。

#127 已在研究/测试边界内完成九领域候选值、路线派生距离布局、完整当前生产内存账本、两类进度候选正确性以及正式多进程性能矩阵。#144 随后消费这些输入完成完整生产候选、内存护栏和五轮稳态矩阵；候选 B 仅提升 `4.257%`，未达到 `5%` 生产门槛，因此 Core/Data/schema 已完整回退 current-f64。

### 2.2 当前范围与规模固定样例

当前 production contract 基本没有物理量上界：`EdgeLength`、`Speed`、`EdgeProgress` 和 profile/Parking fields 接受满足符号约束的有限 `f64`，Core 还有 `f64::MAX` 与累计 overflow 防御测试。首轮候选因此必须分别报告：

- **迁移前实现范围**：对当前无上界 API/schema 的有意行为变化；
- **product envelope**：项目定位要求正式支持的局部道路范围；
- **pathological range**：极值、overflow 和 rejection seeds；
- **scale fixture range**：为容纳 10k/100k vehicles 而生成的合成 edge 长度。

现有代表性 single-edge builders 的关键范围：

| workload                                     |                          10k edge length | 100k edge length | `f32` ULP at 100k | 说明                                        |
| -------------------------------------------- | ---------------------------------------: | ---------------: | ----------------: | ------------------------------------------- |
| dense / stop-and-go platoon，`6.5 m` spacing |                               `66_000 m` |      `651_000 m` |        `0.0625 m` | 车辆密度有代表性，单 edge 长度是规模夹具    |
| free-flow platoon，`250 m` spacing           |                            `2_501_000 m` |   `25_001_000 m` |             `2 m` | ULP 大于 `10 m/s × 16 ms = 0.16 m` 单步位移 |
| command spatial，`10 m` spacing              |                              `101_000 m` |    `1_001_000 m` |        `0.0625 m` | 主要测 command candidate/query scale        |
| transition-heavy                             | 每条 `5 m`，edge 数随 vehicle count 增长 |             同左 | 小于 `0.000001 m` | locality-preserving，适合 boundary/oracle   |

因此 unchanged builders 继续作为迁移前范围/规模探针；另需增加等 vehicle count、等 density、但每条 edge/local coordinate 不超过 product envelope 的 companion workload。后者只改变 fixture data layout，不实现 #72 的 runtime partition、multi-rate 或并行调度。

### 3.1 工作负载分类规则

每个 `f64`/candidate build 都必须运行相同输入的两组矩阵：

1. **legacy/current-range matrix**：保持现有 builders 不变，用于暴露 range narrowing、stalled progress、ordering 与 overflow 行为；
2. **locality-preserving matrix**：vehicle count、profile、density、step/command sequence 与 observation 数保持一致，仅把合成 mega-edge 分解为 envelope 内的多 edge/多 route fixture，用于评估产品范围内的时间、内存和 cache/SIMD 收益。

不得因为 candidate 在 legacy mega-edge 上失败就直接断言 `f32` 不适合 LaneFlow 产品范围，也不得因为 locality-preserving workload 通过就隐瞒 current public range 的有意破坏性变化。两组结论分别进入 ADR 的迁移范围与性能部分。

### 3.3 `f64` 保持局部性的 1 万车辆基线（2026-07-18）

测试/benchmark support 在 commit `a2ca90dc60d99f5d7f1e727dc1295ebb263aa217` 增加 companion builders：vehicle count、spacing、profile、初始状态和 60-step sequence 与 legacy builder 相同，只把 route 分成最大 `10_000 m` 的连续 edge。10k free-flow 使用 251 edges，dense/stop-and-go 各使用 7 edges；没有修改 production scalar、control flow、public API 或 schema。

## 4. 候选实现比较规则

- `f64` 基线必须来自未修改的 production control flow。
- `f32` 或 mixed candidate 必须运行同一 scenario builder、step 数、command 序列和 validation。
- 如使用临时 research-only scalar alias、feature 或完整转换 commit，必须记录 commit/patch；Delivery PR 不得无意保留未接受的 feature 或公共泛型。
- 比较除 wall time 外，还必须覆盖 `VehicleState`、hot scratch buffers、route-distance index、occupancy 和 command spatial index 的 accounted bytes。
- progress candidate 若引入 residual/compensation，residual 本身、padding 和 scratch mirror 必须全部计入 accounted bytes。
- 构造、clone 或 fixture setup 时间与 steady-state step/command 时间分开记录。

## 5. 判定基准与差分合同

同一输入序列下，以 current `f64` 行为作为 reference oracle。候选实现逐 tick 比较：

- tick/time、status、route edge index、event kind/order 等离散结果必须精确一致；
- progress、speed、acceleration、gap 和 constraint distance 按各自领域预算比较；
- `NaN` / Infinity rejection、canonical zero 和错误类别必须一致；
- 失败路径必须保持 world 原子不变；
- boundary 邻域使用 exact boundary、上下一个可表示值和领域 epsilon 两侧的 seeds；
- 覆盖 multi-edge traversal、SignalStop、ParkingStop、RouteEnd、leader change、safe-speed 和 no-overlap projection。

若候选导致离散行为不同，必须记录首次 divergence 的 tick、输入、前后状态和数值差；不得只比较最终位置或平均误差。

### 5.1 重复累加探针

以下 probe 只验证“raw `f32` authority 逐 tick 重复累加”的风险，不替代 production control-flow oracle。每次乘法和加法后都显式 round 到 IEEE binary32；compensated 列使用一个额外 `f32` residual：

| speed / fixed delta / ticks   |   数学位移 |   raw `f32` result |      raw error | compensated result | compensated error |
| ----------------------------- | ---------: | -----------------: | -------------: | -----------------: | ----------------: |
| `10 m/s` / `16 ms` / `6_250`  |  `1_000 m` |    `999.9282837 m` | `-0.0717163 m` |   `1000.0000610 m` |    `+0.0000610 m` |
| `1 m/s` / `16 ms` / `62_500`  |  `1_000 m` |    `999.6366577 m` | `-0.3633423 m` |   `1000.0000610 m` |    `+0.0000610 m` |
| `30 m/s` / `16 ms` / `36_000` | `17_280 m` | `17_288.5664063 m` | `+8.5664063 m` |         `17_280 m` |             `0 m` |

结论仅限于研究路由：静态 ULP 在 local range 内足够，不足以证明 raw repeated addition 满足 long-run progress budget。候选必须比较至少 `raw f32`、`f32 + residual/compensation` 与保留 `f64 progress/integration`；若 residual 使 progress storage 回到 8 bytes，内存结论必须如实体现。

### 6.2 候选接受上限

候选必须同时满足绝对误差、常规非零值的相对误差和离散行为三类条件：

| 数值域                            |                     最大绝对误差 |                      最大相对误差 | 离散行为要求                                              |
| --------------------------------- | -------------------------------: | --------------------------------: | --------------------------------------------------------- |
| edge length                       |                         `0.01 m` |            `1e-6`，value `>= 1 m` | graph/route validation 分类一致                           |
| edge progress                     | `0.01 m`，每 tick 与 boundary 后 |         `1e-5`，progress `>= 1 m` | transition count/order、remainder、completion 精确一致    |
| bumper gap / route-local distance |                         `0.01 m` |       `1e-4`，`abs(value) >= 1 m` | leader、Passed/Within/BeyondHorizon、projection 精确一致  |
| speed                             |                       `0.01 m/s` |          `1e-4`，speed `>= 1 m/s` | stopped/resume、constraint selection 精确一致             |
| acceleration                      |                      `0.02 m/s²` |    `1e-3`，`abs(value) >= 1 m/s²` | comfort/emergency ordering 与 projection attribution 一致 |
| Parking offset/size               |                         `0.01 m` |       `1e-4`，`abs(value) >= 1 m` | bind/arrival/leave 与 validation 分类一致                 |
| local coordinate                  |                         `0.01 m` |       `1e-5`，`abs(value) >= 1 m` | edge binding、sample segment 与 endpoint 结果一致         |
| heading                           |                     `0.0001 rad` | N/A，canonical angle 使用绝对误差 | canonical range、tangent direction 与 wrap 结果一致       |

接近零时不使用相对误差。no-overlap、edge transition、SignalStop、ParkingStop 和 RouteEnd 的离散结果是 hard gate：即使连续误差低于 ceiling，只要这些结果改变，candidate 仍为失败，除非 G1 明确修改行为契约并给出单独迁移依据。

长时段 drift 至少覆盖 60-step representative workload、`36_000` ticks（16 ms 下约 10 分钟）constant-input probe，以及跨多个 edge 后的 residual/remainder 行为；任一连续误差不得无界单调增长。

### 6.3 Epsilon 推导约束

历史 `EDGE_BOUNDARY_EPSILON` 和 `GEOMETRY_GAP_EPSILON` 虽同为 `1e-9`，却混合了不同职责。#125 已把当前 f64 生产代码拆成九个 crate-private（仅 crate 内可见）所有者：最小 edge 长度、最小车辆长度、Parking anchor 端点留白、Parking 横向偏移最小非零绝对值、Parking extent、edge 边界/余量（boundary/remainder）、纵向约束、物理间距/重叠（gap/overlap）与计算速度近零（computed-speed near-zero）。当前值和比较边界保持不变，两个旧公开常量直接删除。

| 语义                                   | 当前混用点                                    | G1 要求                                                  |
| -------------------------------------- | --------------------------------------------- | -------------------------------------------------------- |
| minimum valid edge length              | `EdgeLength` constructor/schema minimum       | 产品数据语义；不得由 arithmetic tolerance 顺带决定       |
| edge boundary snap tolerance           | progress、remainder、transition/arrival       | 由 max ULP、运算链与 `0.01 m` ceiling 推导               |
| minimum geometry extent/nonzero offset | profile/Parking length、width、lateral offset | 产品几何语义；与 overlap tolerance 分离                  |
| gap/overlap zero tolerance             | leader gap、no-overlap、projection            | 由两个 progress 的量化/相减和 hard safety invariant 推导 |
| heading canonicalization tolerance     | #123 sampling/normalization                   | 由 angle ULP 与 `0.0001 rad` ceiling 推导                |

首轮使用 `4 * max ULP` 作为 add/subtract/snap 的保守 representation floor，而不是最终生产常量。在 `16_384 m` progress 上该 floor 为 `0.0078125 m`，低于但已接近 `0.01 m` ceiling；这意味着 raw local `f32` 的可用裕量很窄。候选可通过缩小 guaranteed edge/local range、保留 `f64` authority、使用 residual/compensation 或重新设计 boundary calculation 解决，但必须由 oracle 证明。相对误差和 ULP 只允许用于 #127 离线标定；production 只消费固定、带单位、领域化的绝对阈值和领域专用 predicate。

## 7. `f16` / 量化矩阵

Rust 1.96 的 native `f16` 仍不稳定；研究使用 `half 2.7.1` 的 IEEE binary16 conversion。该 crate 为 `MIT OR Apache-2.0`、MSRV 1.81，本轮只加入 dev-dependency；cargo-deny 0.20.2 的 advisories/bans/licenses/sources 全部通过。`half` 文档说明算术可能经 f32 模拟，因此本轮只评价 storage/round-trip，不把它当作 runtime arithmetic 加速证据。

IEEE binary16 在 envelope 内遍历相邻有限值后的 worst half-step：

| Domain          |           Range | binary16 max round-trip error |      Ceiling | Result                                |
| --------------- | --------------: | ----------------------------: | -----------: | ------------------------------------- |
| progress        |  `0..=16_384 m` |                         `4 m` |     `0.01 m` | Fail                                  |
| speed           |   `0..=100 m/s` |                 `0.03125 m/s` |   `0.01 m/s` | Fail                                  |
| acceleration    | `-50..=50 m/s²` |               `0.015625 m/s²` |  `0.02 m/s²` | 单域 Pass，但不足以承担控制 authority |
| extent / offset |  `-128..=128 m` |                   `0.03125 m` |     `0.01 m` | Fail                                  |
| heading         |      `-PI..=PI` |            `0.0009765625 rad` | `0.0001 rad` | Fail                                  |

领域专用整数定点可以满足 round-trip ceiling，但会引入新的 range/saturation/wire contract：

| Domain / format                   |    Representable range |        max error |      Ceiling | Result            |
| --------------------------------- | ---------------------: | ---------------: | -----------: | ----------------- |
| progress `u32` centimeters        |   `0..42,949,672.95 m` |        `0.005 m` |     `0.01 m` | quantization Pass |
| speed `u16` centimeters/s         |        `0..655.35 m/s` |      `0.005 m/s` |   `0.01 m/s` | quantization Pass |
| acceleration `i16` centimeters/s² | `-327.68..327.67 m/s²` |     `0.005 m/s²` |  `0.02 m/s²` | quantization Pass |
| heading `u16` turn                |            `[-PI, PI)` | `0.00004794 rad` | `0.0001 rad` | quantization Pass |

因此 `f16` 禁止用于 Core/Spatial authority；整数定点也不作为 current API/schema 的隐式替换。它们只可在未来的 versioned storage/transport、GPU buffer 或可丢弃 distant presentation cache 中按用途单独设计，必须携带明确 scale/origin、checked overflow、rounding 与 non-authoritative 声明。

### 9.2 被修正的候选语义

旧 `compensated_f32` 只在 `progress += delta` 时保存 Kahan 风格 residual，却在以下位置继续只读取 high component：

- 同 edge 的 leader/follower progress 差；
- 跨 edge 的 remaining distance 与 leader progress；
- edge boundary 判断和归零；
- snapshot 与差分报告。

因此 residual 只是“写时存在、读时丢失”的隐藏状态，不能代表完整的 compensated progress。#140 新增 `residual_aware_f32`，将有效进度定义为 `high - residual`：

```text
same-edge distance
  = (leader_high - follower_high)
  - (leader_residual - follower_residual)

remaining distance
  = (edge_length - progress_high) + progress_residual
```

实现时先把每个 high/residual 分量逐一提升为 f64，再执行减法并转换为 compute scalar，避免 residual 或非近邻操作数先产生不必要的 f32 中间舍入。这不表示所有 high 分量的 f32 减法都会丢位：同号且足够接近的 binary32 操作数在 Sterbenz 条件下可以精确相减；逐分量提升的意义是让实现不依赖这些前提，并让 f64 敏感对照保留完整组合语义。本轮修订前后的已报告误差量级与通过结论一致。`sensitive_f64_control_f32_progress` 使用同一 progress 表示，但把 motion/profile/IIDM/safe-speed/projection compute 与 speed state 升宽为 f64；它用于判断是否真的需要升宽敏感控制链。

### 9.8 研究判断与后续治理

1. `10_000 m` 已由 #141 接受为下一契约的单 edge 防御性硬上限，而不是日常建模目标长度。edge 应代表拓扑、通行规则、控制和几何连续的有向车道段；连接关系、Signal/Parking 控制点、车道属性或几何不连续处本就应拆分。数百公里道路应由 route 组合多个 edge，不应靠一个超长 edge 表示。
2. 10 km 的二进制 32 位浮点数（binary32）ULP 为 `0.0009765625 m`，四个 ULP 为 `0.00390625 m`，低于 `0.01 m` 进度上限；相比 `16_384 m` 的 `0.0078125 m` 四 ULP 下限留有更合理裕量。4 km/1 km 没有必要仅为数值精度成为硬上限；100 m 虽在本工作负载更快，却会显著增加创作 ID、route 出现位置和静态拓扑，不应由微基准决定。
3. 当前生产 API/模式接近接受全部有限 `f64`。增加 10 km 硬上限属于有意的行为和 Data 验证变化；#126 冻结目标版本 `formatVersion: "0.6"` 并把离线标定交给 #127、原子候选交给 #144。#144 no-go 后当前版本仍是 v0.5。
4. 补偿残差感知的 `f32` 已成为 ADR 0014 接受的目标方案；它已经通过当前研究模型的严格连续预算、离散硬闸口、同运行时重放、长时累计和 1 万/10 万车辆性能与内存观察。它仍未覆盖完整生产 Signals、Parking、route 距离、命令空间索引、公共错误/API/Data 规范化，因此当前 Core 不在 #141 文档 PR 中切换。
5. 不建议“敏感控制链遇到误差就升宽到 `f64`”。当前误差根因已由正确的高位/残差读取语义解决；额外的 `f64` 控制只提供没有被预算要求的精度，并失去候选内存与吞吐优势。
6. ADR 0014 已取代 ADR 0012 的目标数值决策；ADR 0012 继续描述迁移前的生产 `f64` 实现状态。目标已接受不等于生产已切换。
7. #125 保持当前 `f64` 行为不变并拆分九个领域所有者；#126 冻结 Core 公开表面、Parking 锚点的 `EdgeProgress`、分层错误载荷和 Data v0.6 原子迁移矩阵；#127 使用真实目标 `f32` 运算链完成离线标定；#144 完成原子生产候选后因性能门槛失败而回退；#128 应按该 no-go 事实收口。

## 11. #127 完整保留内存账本基线

### 11.1 统计口径与生命周期阶段

源码基线为提交 `bc6bf7c852a7a25019078330d9c271bf16cc5441`。仅用于测试的保留内存核算器（test-only accountant）现在枚举 `CoreWorld` 的全部堆所有者：车道图（LaneGraph）、车辆配置（Vehicle Profile）注册表、静态信号（Signals）注册表、已提交信号状态、信号候选暂存区、路线及距离/反向引用、车辆槽位与解析表（resolver）、生命周期暂存区、停车（Parking）、占用、纵向暂存区和命令空间索引。

完整核算值使用以下唯一所有权规则：

```text
完整核算字节（complete_accounted_bytes）
  = CoreWorld 内联大小（size_of::<CoreWorld>()）
  + 每个唯一拥有的堆容量与嵌套字符串容量之和
```

- `CoreWorld` 的内联大小只计一次；它已经包含各子所有者的容器头（header）和内联填充（padding）。诊断同时输出主要所有者的 `size_of`，但不再次加入总数。
- `Vec`、`BinaryHeap` 和 `IndexMap` 的条目缓冲区（entry buffer）按公开容量（capacity）乘实际元素 `size_of`；元素填充随 `size_of` 一并计入。嵌套 `String`、内层 `Vec` 和每条路线/索引缓冲区分别计一次。
- 清理车辆时，`IndexMap::capacity()` 会受随机哈希墓碑槽（tombstone）分布影响，不能代表后备存储（backing allocation）是否释放。矩阵因此冻结清理前解析表的后备存储字节，只按仍存活的字符串键容量调整清理后值；1 万规模连续两轮结果逐字节一致。
- 该模型用于比较同一依赖、同一分配器（allocator）和同一工作负载下的 Core 布局；它不是进程常驻集大小（RSS），不计分配器元数据、代码页、测试夹具（fixture）临时值和操作系统页粒度。

三个阶段定义如下：

1. 当前（`current`）：完成世界构造、尚未执行逻辑步（tick）；包含权威状态和构造时已建立的命令空间索引。
2. 高水位（`high-water`）：执行一个 16 ms 预热逻辑步，并在仍保留全部车辆字符串和槽位时建立命令移除所需的精确速度堆；再与清理检查点比较，保留完整核算值最大的快照。
3. 清理后（`cleaned`）：依次移除全部车辆后统计；静态图、路线、信号、停车和可复用暂存区/索引容量继续保留。

## 12. #127 分层数值契约标定与最小 edge 选择

### 12.1 证据边界、环境与复现命令

本节的标定器来自提交 `9460286`，最小 edge 压力基准来自提交 `976ad74`。主验证工具链仍为 Rust 1.96.0；机器与电源模式沿用 11.3 节。固定种子为 `0x1270014001410144`。研究代码只存在于测试与基准目录，没有修改生产标量、公开 API、Data 格式或 `numeric_policy` 中 current-f64 的九个 `1.0e-9` 值；#144 曾在真实生产布局中投入本节候选，但 no-go 后已回退。

```powershell
cargo +1.96.0 test -p laneflow-core --test minimum_edge_research --locked -- --nocapture

```

转换判定基准（conversion oracle）固定以下顺序：

1. 在原始 `f64` 上检查有限性和产品范围，错误保留原始输入与阶段；
2. 执行受检 `f32` 转换，拒绝溢出、非有限目标值和转换后失去领域有效性的下溢；
3. 把正负零统一为正零；
4. 再对规范化目标权威值执行产品范围复核、跨字段顺序、edge 绑定与 Parking anchor 校验；
5. 批量输入先全部转换到暂存向量，任一元素失败时不修改权威向量。

这一区分防止 `0.1_f64.next_down()`、`128_f64.next_up()` 或 `10_000_f64.next_up()` 在先转 `f32` 后错误落回合法边界，也防止极小的非零 Parking 横向偏移静默变成零。

`1.0 m` 是不含边界（exclusive）下限，因此目标 f32 的首个合法值为 `1.0_f32.next_up()`，即约 `1.0000001192 m`。仓库中现有 `1.0 m` edge 只出现在句柄（handle）与机械路线跟随（route-following）测试；Signals、Parking 和车辆跟驰固定样例的最小实际 edge 为 `5.0 m`，另有一个 `2.0 m` route 终段。#144 候选曾把纯机械 `1.0 m` 样例改成合法目标值，no-go 后随生产语义一起回退。edge 表达拓扑、规则和控制连续的车道段，不是几何曲线细分；Parking 端点由同一 edge 内的 anchor 表达，因此没有发现要求小于等于 1 m Core edge 的生产用例。

### 12.3 四类运行时算术候选

运行时判定基准（runtime oracle）让高精度参照与补偿残差感知 `f32` 候选从同一份规范化目标权威值出发。相对误差和末位单位（ULP）只出现在本离线报告；生产候选仍是固定、带单位、领域专用的绝对值。

| 算术语义所有者                           |   固定绝对候选 |    样本数 |     实测最大绝对误差 | 候选/实测 | 主要运算链与余量                                                                                                                 |
| ---------------------------------------- | -------------: | --------: | -------------------: | --------: | -------------------------------------------------------------------------------------------------------------------------------- |
| edge 边界/余量（boundary/remainder）     | `0.00000001 m` | `540,059` |                `0 m` |     `inf` | 有效进度加 f32 travel；跨 edge 后用 f64 有效余量重新拆为高位/残差（high/residual）；候选还负责让精确零命中严格小于（strict）判定 |
| 纵向约束                                 |    `0.00005 m` | `200,217` |   `0.000007629395 m` |  `6.5536` | 1/16/1000 ms、弹道 travel、位置匹配和派生速度到距离；为多次乘加与最终加法保留保守余量                                            |
| 物理间距/重叠（gap/overlap）             |    `0.00001 m` | `100,252` |   `0.000007080078 m` |  `1.4124` | 同 edge/跨 edge 有效进度先在 f64 消去大坐标，再转局部前距并减 `f32` 车辆长度                                                     |
| 计算速度近零（computed-speed near-zero） |  `0.00005 m/s` | `100,243` | `0.000008285046 m/s` |  `6.0350` | `candidate_speed - 2 * removed_travel / delta_time`，覆盖最小 1 ms 放大和后续除、乘、减舍入                                      |

edge 候选不是从“最大坐标的四个 ULP”推导。重基线后，所有边界算术样本都能从两个 `f32` 分量重建为与高精度链相同的有效余量；`1e-8 m` 主要保留余量严格小于判定对精确零的包含语义，并低于 `5e-5 m/s` 近零速度在最小 tick 下的最小弹道行程量级。纵向约束与计算速度近零使用 `5e-5`，但它们仍是两个独立所有者：前者单位为米并控制约束归因，后者单位为米每秒且承担除以 tick 的放大链。物理间距可收紧到 `1e-5 m`，因为接触邻域的局部前距不超过车辆长度上界附近；最后两个同量级 `f32` 的相减满足 Sterbenz 精确减法条件，主要误差来自前距转换。

初版研究候选在跨 edge 时调用普通补偿加法加入 `-edge_length`。大负增量会在 `delta - residual` 中吞掉很小的残差：`128 m` edge、`13.9 m/tick`、36,000 tick 的失败样本累计到 `0.007722854614 m`，已经逼近 `0.01 m` 总预算。把 epsilon 放大到 1 cm 只会掩盖问题。新链先读取 `f64(high) - f64(residual)`（高位减残差），减去规范化 `f32 EdgeLength`，再把真实余量重新拆成高位/残差（high/residual）；同一矩阵的最大误差降为零。#144 候选实现并复跑了该算法；它随 no-go 候选回退，不是当前生产行为。

### 12.4 离散判定、归因与失败原子性

规范化高精度参照与补偿残差感知 `f32` 候选运行 4 种合法 edge 布局、3 类车队场景、60 个 tick、每组 256 辆车。edge index、状态、前车（leader）、投影与 step 事件摘要零分歧；连续值如下：

| 连续值               |          实测最大误差 |    接受上限 | 结果 |
| -------------------- | --------------------: | ----------: | ---- |
| route progress       |    `0.000002160668 m` |    `0.01 m` | Pass |
| bumper gap           |    `0.000001616776 m` |    `0.01 m` | Pass |
| speed                |  `0.000037910029 m/s` |  `0.01 m/s` | Pass |
| applied acceleration | `0.002500941902 m/s²` | `0.02 m/s²` | Pass |
| final travel         |    `0.000001490116 m` |    `0.01 m` | Pass |

RouteEnd、SignalStop、ParkingStop 与跟车硬投影（hard projection）的小型归并矩阵覆盖 1/16/1000 ms、精确边界（exact）、`next_down`、`next_up`、阈值两侧、单约束、相同距离平局和前车二次收紧，共 180 组；离散分歧为零。相同 travel/speed 时保持信号、停车、路线终点、基础候选（Signal、Parking、RouteEnd、Base）的既有优先级，空间约束投影事件继续先于同车的跟车投影事件。#144 已在真实 Signals/Parking/RouteEnd 调用链上复跑并通过，但性能 no-go 后没有进入生产。

转换回归另外锁定：`NaN`/Infinity 原始阶段拒绝、`f64::MAX -> f32` 溢出阶段拒绝、正负零规范化、Parking offset 下溢后拒绝，以及三元素批量在中间失败时权威向量逐值不变。current-f64 的逻辑步、命令、加载（step/command/load）原子性继续由现有生产测试覆盖；#144 曾验证目标 f32 的真实结构化错误和 world/data 提交边界，no-go 后当前错误表面仍为 f64。

### 12.5 最小 edge 压力与选择结论

最坏输入使用目标硬上限 `100 m/s` 与 `1,000 ms`，即单 tick 约 `100 m` 行程。计数按每个不含边界下限的首个合法 `f32` edge 长度计算。事件内存下限只乘当前 `size_of::<CoreEvent>() = 64 B`，不含 `Vec` 容量、路线、Signals/Parking 检查或其他暂存区，因此只能低估完整成本。

|          最小 edge 不含边界候选 |            首个合法 f32 edge |   每车跨界 |     10k 跨界 |     100k 跨界 |  100k 事件字节下限 |
| ------------------------------: | ---------------------------: | ---------: | -----------: | ------------: | -----------------: |
| 精确正数（exact-positive）`0 m` | 最小正次正规（subnormal）f32 | 超过 `u64` |     无有界值 |      无有界值 |           无有界值 |
|                        `0.01 m` |              `0.010000001 m` |    `9,999` | `99,990,000` | `999,900,000` | `63,993,600,000 B` |
|                         `0.1 m` |                      `0.1 m` |      `999` |  `9,990,000` |  `99,900,000` |  `6,393,600,000 B` |
|                         `1.0 m` |                `1.0000001 m` |       `99` |    `990,000` |   `9,900,000` |    `633,600,000 B` |

`release`（发布优化）紧凑内核逐次执行了全部比较与相减，没有用除法公式替代循环，也没有分配事件。带 17 相位扰动后的实际计数为：10k 下约 `99,994,116 / 10,000,000 / 999,411`，100k 下约 `999,941,174 / 100,000,000 / 9,994,117`，方向与静态下限一致。单次聚合运行不是性能服务级目标（SLA）；正式墙钟时间（wall-time）仍按 G1 协议使用 `minimum_edge_candidates` Criterion 性能基准，在相同省电模式下运行至少五个独立进程轮次。

结论：选择 `1.0 m` 不含边界下限。它不是因为 `f32` 无法表达更短 edge，而是因为 Core edge 是会产生拓扑、事件、控制检查和失败原子性成本的产品对象；0.01/0.1 m 会把最坏每车循环扩大到 9,999/999 次，并产生数量级不可接受的事件下限。1 m 仍允许 2 m 终段、5 m 短连接和所有当前 Signals/Parking/跟驰固定用例；更细几何应留在 Spatial 曲线和采样层。#144 曾实施 Core/Data v0.6 的受检范围与错误迁移，但 no-go 后该判断仍只是未来迁移输入。

## 13. #127 路线派生距离布局选择

### 13.1 候选边界、正确性与复杂度

候选实现来自提交 `d106694`，满足交替顺序协议的性能夹具来自提交 `6466292`。两个候选都只从已经规范化的目标 `f32 EdgeLength` 派生，不保存原始输入的第二份长度权威，也不修改当前生产 `RouteDistanceIndex`。基础复现命令为：

```powershell
cargo +1.96.0 test -p laneflow-core --test route_distance_candidates --locked -- --nocapture
cargo +1.96.0 clippy -p laneflow-core --test route_distance_candidates --bench route_distance_candidates --locked -- -D warnings
cargo +1.96.0 bench -p laneflow-core --bench route_distance_candidates --locked --no-run
```

候选使用以下布局：

1. `f64` 前缀基线为每个路线出现位置保存段内 `f64` 偏移。按 `10,000 m` edge 上界推导，每段最多 `3,518,437,208` 个出现位置，使段长不超过 `2^45 m`；该量级的 `f64` ULP 为 `0.0078125 m`，仍低于 `0.01 m` 连续预算。出现位置直接用整数除法定位段，查询为 O(1)，不需要二分搜索或扫描路线。
2. 分块局部 `f32` 候选每 6 个出现位置形成一块，每个出现位置保存块内 `f32` 偏移，每块保存 `f64` 段内基值和总长。最坏块内偏移不超过 `50,000 m`；段最多 `586,406,201` 块，同样保持段长不超过 `2^45 m`，并用整数除法 O(1) 定位。
3. 两个候选的稀疏段前缀都使用双分量 `f64` 累加。它只保护超长路线后段的局部差值，不改变公开距离类型，也不引入第二份 edge 长度。`1e23 m` 前缀后的 `12,345.678901234 m` 局部差值恢复误差不超过 `1e-9 m`。
4. 到路线终点的后缀距离继续保留每出现位置的有界 `f64` 派生值；因此两种候选都保持 RouteEnd 的 O(1) 查询和现有超视距分类。

固定种子 `0x127_0014_0141_0127` 生成 4,096 个合法目标 edge 和 100,000 次混合查询，并覆盖同位置后退、有限视距内外、精确边界、`next_down`、强制跨段和巨大前缀消去。结果如下：

| 候选           | 最大路线距离绝对误差 | 离散分类分歧 | O(1) 查询 | 结果 |
| -------------- | -------------------: | -----------: | :-------: | ---- |
| `f64` 前缀     |                `0 m` |          `0` |    是     | Pass |
| 分块局部 `f32` |   `0.002929568291 m` |          `0` |    是     | Pass |

分块误差低于 `0.01 m`，所以正确性本身不能淘汰该候选；布局选择还必须继续比较完整内存贡献与执行性能。

### 13.4 选择与 #144 实施边界

#127 选择 `f64` 前缀基线，综合依据为：

- 两者正确性、离散行为和 O(1) 复杂度都通过；
- 分块在路线密集索引内节省约 `5.39%`，但在路线密集完整目标布局中只额外贡献约 `0.92%`–`0.98%`，没有达到 G1 冻结的 `10%` 完整内存胜出条件；
- 分块冷构造配对中位数快约 `26.20%`，但运行时查询有 5/6 轮稳定回退，配对中位数慢约 `26.06%`，独立性能前置门槛失败；
- 分块还增加局部偏移、块基值/总长和两级定位的不变量，收益不足以支付热路径和审阅复杂度。

#144 候选实现了本节胜出的“规范化 `f32 EdgeLength` 精确提升后形成的分段局部 `f64` 前缀”布局，并在真实 Parking、Signals、RouteEnd 和命令空间索引上复跑正确性与完整内存矩阵。整体稳态性能 no-go 后，该布局随目标 `f32 EdgeLength` 一起回退；超长路线的分段、防消去和超视距研究语义仍作为未来输入保留。

### 14.4 正确性、长时与失败矩阵

### 14.5 候选选择与 #144 生产规则

#127 可以冻结以下结论：

1. 残差感知 `f32` 已通过自身正确性和性能门槛；不能再以旧 high-only compensated 候选的失败否定它。
2. `f32` 计算 + `f64` 进度在同一正确性矩阵得到完全相同的误差和离散结果，在研究布局中也与残差候选完全同为 `32 B` vehicle、`56 B` motion、`88 B/vehicle`；1 万/10 万分别都是 `880,104 B` / `8,800,104 B`。
3. 混合进度候选在六轮主矩阵和八轮跨 edge 矩阵中持续快于残差候选。残差表示没有表现出可重复的吞吐或研究布局内存收益，因此不能在 #127 中被宣称为已经胜出的唯一生产布局。
4. #127 没有实现完整目标生产布局，所以研究模型的 `88 B/vehicle` 仍不能证明两者在真实 CoreWorld 的完整保留内存绝对相同；#144 后续实测候选 A/B 的 36 项账本字节级相同，并把旧 10% 强制收益门槛修订为内存退化护栏。

因此 ADR 0014 第 9.3 节的回退条款现在成为 #144 的强制选择规则：在同一个原子迁移 PR 内提供两种仅供验证的目标布局，使用相同真实生产正确性、完整保留内存和性能矩阵比较；若正确性与完整内存相同，生产应选择更简单且更快的单个 `f64` EdgeProgress。只有残差感知布局在完整生产账本中证明了混合布局没有的实质收益，才可继续作为最终权威。比较完成后只能保留一个生产布局，不增加运行时精度开关、双重权威或兼容层。

其他冻结输入不变：

- 九领域固定候选值继续采用第 12 节结果；#144 候选曾一次性启用，no-go 回退后当前生产值不变；
- route 派生距离采用第 13 节胜出的“规范化 `f32 EdgeLength` 精确提升后形成的分段局部 `f64` 前缀”；
- 当前完整生产保留内存账本采用第 11 节六场景基线；#144 已转换完整目标布局并复测 1 万/10 万、预热/峰值/清理后阶段，36 项均通过修订后的内存护栏；
- `half 2.7.1` 继续只作为研究/测试开发依赖，用于非 ignored 的 binary16 量化回归；它不是生产算术候选。许可证、MSRV、RustSec 和来源继续由 cargo-deny 覆盖；
- #127 的性能通过不等于 #144 自动通过。#144 的真实 Signal/Parking/route、公开错误和 Traffic Data v0.6 候选通过正确性，但最终稳态性能门槛失败，生产没有切换。

### 15.2 正确性与完整内存

候选 A/B 都通过 Rust 1.96 工作区测试、Clippy `-D warnings` 和生产镜像预言机。候选 B 在 edge 端点出现约 `-6.98e-10 m` 的舍入余量，按 #127 已冻结的 `1e-8 m` edge-boundary 领域判定在重基线前归零；没有放宽阈值。

候选 A/B 的六场景 × 两规模 × 三生命周期阶段共 36 项完整保留内存记录字节级相同。当前态变化范围为增加 `1.700%` 到减少 `5.183%`，全部低于 `5%` 回退护栏，也没有增加超过 `2%` 的待解释记录。内存因此通过护栏，但不决定进度布局。

### 15.5 生产结论

结果是 **no-go**，不是“接近门槛所以酌情通过”：

1. 候选 A 已因性能和复杂度淘汰；
2. 候选 B 通过正确性与内存护栏，但主稳态性能只提升 `4.257%`；
3. 构造、命令受检转换和 route-distance 查询属于次级矩阵，不能抵消主门槛失败，因此停止执行；
4. 生产 Core、Data、schema、样例和九领域阈值完整恢复 current-f64，不保留部分 `f32` 权威、运行时精度开关、v0.6 加载分支或候选专用热路径错误布局；
5. 通用 production benchmark 和同形场景夹具保留，供未来候选按同一口径复测。

未来若重启数值迁移，必须新建议题并重新进入 G1，说明相对 #144 能跨过收益门槛的新实现或新硬件证据；不得直接恢复实验提交，也不得降低已确认的正确性预算来换取通过。
