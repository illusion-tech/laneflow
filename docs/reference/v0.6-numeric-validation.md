
结论：当前 scalar 迁移至少同时触及 public Core newtype/state/error、Data normalization 范围、per-vehicle hot state、retained route/command indexes 和 Parking public geometry。JSON token 仍可保持 `number`，但只要 loader 可接受范围、rounding 或 normalization 语义改变，就必须按 ADR 0008 评估 current format version，而不能仅以 JSON shape 未变为由跳过 data-format 影响。

### 2.2 Current range 与 scale fixture

当前 production contract 基本没有物理量上界：`EdgeLength`、`Speed`、`EdgeProgress` 和 profile/Parking fields 接受满足符号约束的有限 `f64`，Core 还有 `f64::MAX` 与累计 overflow 防御测试。首轮候选因此必须分别报告：

- **current compatibility**：对现有无上界 API/schema 的行为变化；
- **product envelope**：项目定位要求正式支持的局部道路范围；
- **pathological range**：极值、overflow 和 rejection seeds；
- **scale fixture range**：为容纳 10k/100k vehicles 而生成的合成 edge 长度。

现有代表性 single-edge builders 的关键范围：

| workload | 10k edge length | 100k edge length | `f32` ULP at 100k | 说明 |
| --- | ---: | ---: | ---: | --- |
| dense / stop-and-go platoon，`6.5 m` spacing | `66_000 m` | `651_000 m` | `0.0625 m` | 车辆密度有代表性，单 edge 长度是规模夹具 |
| free-flow platoon，`250 m` spacing | `2_501_000 m` | `25_001_000 m` | `2 m` | ULP 大于 `10 m/s × 16 ms = 0.16 m` 单步位移 |
| command spatial，`10 m` spacing | `101_000 m` | `1_001_000 m` | `0.0625 m` | 主要测 command candidate/query scale |
| transition-heavy | 每条 `5 m`，edge 数随 vehicle count 增长 | 同左 | 小于 `0.000001 m` | locality-preserving，适合 boundary/oracle |

因此 unchanged builders 继续作为 current compatibility/scale probe；另需增加等 vehicle count、等 density、但每条 edge/local coordinate 不超过 product envelope 的 companion workload。后者只改变 fixture data layout，不实现 #72 的 runtime partition、multi-rate 或并行调度。

## 3. 代表性 workload 合同

性能比较必须在同一机器、Rust 1.96、release profile、锁定依赖和相同 scenario 输入下运行。至少复用：

### `core_step`

- vehicle following free-flow / dense platoon / stop-and-go，10k × 60 steps；
- no-overlap projection，10k；
- repeated edge transition，10k × 60 steps；
- Signals mixed offsets，10k × 60 steps；
- static/activation/occupied Parking，10k × 60 steps；
- 对应 100k observation workload。

### `core_commands`

- fixed 10k world command batches；
- 10k matched batch、repeated route 与 compaction threshold；
- 100k matched/scale observation；
- Parking reserve/cancel、fastest-first/leave 与 parked spawn/despawn。

运行入口：

```powershell
cargo +1.96.0 bench -p laneflow-core --bench core_step --locked
cargo +1.96.0 bench -p laneflow-core --bench core_commands --locked
$env:LANEFLOW_BENCH_100K = '1'
cargo +1.96.0 bench -p laneflow-core --bench core_step --locked
cargo +1.96.0 bench -p laneflow-core --bench core_commands --locked
Remove-Item Env:LANEFLOW_BENCH_100K
```

必须记录：CPU、操作系统、Rust/Cargo 版本、commit、workload、sample count、median/interval、峰值或 accounted memory，以及相对 `f64` 基线的变化。单独的加法/乘法 microbenchmark 不能作为迁移依据。

### 3.1 Workload 分类规则

每个 `f64`/candidate build 都必须运行相同输入的两组矩阵：

1. **legacy/current-range matrix**：保持现有 builders 不变，用于暴露 range narrowing、stalled progress、ordering 与 overflow 行为；
2. **locality-preserving matrix**：vehicle count、profile、density、step/command sequence 与 observation 数保持一致，仅把合成 mega-edge 分解为 envelope 内的多 edge/多 route fixture，用于评估产品范围内的时间、内存和 cache/SIMD 收益。

不得因为 candidate 在 legacy mega-edge 上失败就直接断言 `f32` 不适合 LaneFlow 产品范围，也不得因为 locality-preserving workload 通过就隐瞒 current public range 的 breaking change。两组结论分别进入 ADR 的 compatibility 与 performance 部分。

## 4. 候选实现比较规则

- `f64` 基线必须来自未修改的 production control flow。
- `f32` 或 mixed candidate 必须运行同一 scenario builder、step 数、command 序列和 validation。
- 如使用临时 research-only scalar alias、feature 或完整转换 commit，必须记录 commit/patch；Delivery PR 不得无意保留未接受的 feature 或公共泛型。
- 比较除 wall time 外，还必须覆盖 `VehicleState`、hot scratch buffers、route-distance index、occupancy 和 command spatial index 的 accounted bytes。
- progress candidate 若引入 residual/compensation，residual 本身、padding 和 scratch mirror 必须全部计入 accounted bytes。
- 构造、clone 或 fixture setup 时间与 steady-state step/command 时间分开记录。

## 5. Oracle 与差分合同

同一输入序列下，以 current `f64` 行为作为 reference oracle。候选实现逐 tick 比较：

- tick/time、status、route edge index、event kind/order 等离散结果必须精确一致；
- progress、speed、acceleration、gap 和 constraint distance 按各自领域预算比较；
- `NaN` / Infinity rejection、canonical zero 和错误类别必须一致；
- 失败路径必须保持 world 原子不变；
- boundary 邻域使用 exact boundary、上下一个可表示值和领域 epsilon 两侧的 seeds；
- 覆盖 multi-edge traversal、SignalStop、ParkingStop、RouteEnd、leader change、safe-speed 和 no-overlap projection。

若候选导致离散行为不同，必须记录首次 divergence 的 tick、输入、前后状态和数值差；不得只比较最终位置或平均误差。

### 5.1 Repeated-addition probe

以下 probe 只验证“raw `f32` authority 逐 tick 重复累加”的风险，不替代 production control-flow oracle。每次乘法和加法后都显式 round 到 IEEE binary32；compensated 列使用一个额外 `f32` residual：

| speed / fixed delta / ticks | 数学位移 | raw `f32` result | raw error | compensated result | compensated error |
| --- | ---: | ---: | ---: | ---: | ---: |
| `10 m/s` / `16 ms` / `6_250` | `1_000 m` | `999.9282837 m` | `-0.0717163 m` | `1000.0000610 m` | `+0.0000610 m` |
| `1 m/s` / `16 ms` / `62_500` | `1_000 m` | `999.6366577 m` | `-0.3633423 m` | `1000.0000610 m` | `+0.0000610 m` |
| `30 m/s` / `16 ms` / `36_000` | `17_280 m` | `17_288.5664063 m` | `+8.5664063 m` | `17_280 m` | `0 m` |

结论仅限于研究路由：静态 ULP 在 local range 内足够，不足以证明 raw repeated addition 满足 long-run progress budget。候选必须比较至少 `raw f32`、`f32 + residual/compensation` 与保留 `f64 progress/integration`；若 residual 使 progress storage 回到 8 bytes，内存结论必须如实体现。

## 6. 误差预算

### 6.1 Draft product validation envelope

本表是 #122 candidate 的首轮必测范围，不是已接受的 API/schema hard limit：

| 数值域 | Draft 测试范围 | `f32` max-value ULP | pathological / compatibility 追加范围 |
| --- | --- | ---: | --- |
| edge length | `0.1..=16_384 m` | `0.001953125 m` | current finite `f64`、`f64::MAX`、mega-edge builders |
| edge progress | `0..=edge_length` | `0.001953125 m` | exact boundary、adjacent value、epsilon 两侧、mega-edge stalled-step |
| bumper gap / route-local horizon | `-128..=8_192 m` | `0.0009765625 m` | overlap seeds、Passed/BeyondHorizon、huge/tiny segment mix |
| speed | `0..=100 m/s` | `0.00000762939453125 m/s` | non-finite、negative、current extreme finite |
| acceleration / deceleration | `-50..=50 m/s^2` | `0.000003814697265625 m/s^2` | non-finite、profile ordering、current extreme finite |
| profile/Parking extent、offset、min gap | offset `-128..=128 m`；extent `0.1..=128 m`；min gap `0..=128 m` | `0.0000152587890625 m` | zero/epsilon/negative rejection、current extreme finite |
| time headway | `(0, 60] s` | `0.000003814697265625 s` | zero/negative/non-finite rejection |
| local coordinate | `-16_384..=16_384 m` | `0.001953125 m` | #123 origin/rebase 与超范围诊断待补 |
| heading | `[-PI, PI)` | `0.0000002384185791015625 rad` | `-PI`、`PI`、signed zero、wrap 邻域 |

ULP 使用相邻 IEEE binary32 值复算，例如：

```powershell
$value = [single]16384.0
$bits = [BitConverter]::SingleToInt32Bits($value)
$next = [BitConverter]::Int32BitsToSingle($bits + 1)
[double]$next - [double]$value
```

### 6.2 Candidate acceptance ceiling

候选必须同时满足绝对误差、常规非零值的相对误差和离散行为三类条件：

| 数值域 | 最大绝对误差 | 最大相对误差 | 离散行为要求 |
| --- | ---: | ---: | --- |
| edge length | `0.01 m` | `1e-6`，value `>= 1 m` | graph/route validation 分类一致 |
| edge progress | `0.01 m`，每 tick 与 boundary 后 | `1e-5`，progress `>= 1 m` | transition count/order、remainder、completion 精确一致 |
| bumper gap / route-local distance | `0.01 m` | `1e-4`，`abs(value) >= 1 m` | leader、Passed/Within/BeyondHorizon、projection 精确一致 |
| speed | `0.01 m/s` | `1e-4`，speed `>= 1 m/s` | stopped/resume、constraint selection 精确一致 |
| acceleration | `0.02 m/s^2` | `1e-3`，`abs(value) >= 1 m/s^2` | comfort/emergency ordering 与 projection attribution 一致 |
| Parking offset/size | `0.01 m` | `1e-4`，`abs(value) >= 1 m` | bind/arrival/leave 与 validation 分类一致 |
| local coordinate | `0.01 m` | `1e-5`，`abs(value) >= 1 m` | edge binding、sample segment 与 endpoint 结果一致 |
| heading | `0.0001 rad` | N/A，canonical angle 使用绝对误差 | canonical range、tangent direction 与 wrap 结果一致 |

接近零时不使用相对误差。no-overlap、edge transition、SignalStop、ParkingStop 和 RouteEnd 的离散结果是 hard gate：即使连续误差低于 ceiling，只要这些结果改变，candidate 仍为失败，除非 G1 明确修改行为契约并给出单独迁移依据。

长时段 drift 至少覆盖 60-step representative workload、`36_000` ticks（16 ms 下约 10 分钟）constant-input probe，以及跨多个 edge 后的 residual/remainder 行为；任一连续误差不得无界单调增长。

### 6.3 Epsilon 推导约束

当前 `EDGE_BOUNDARY_EPSILON` 和 `GEOMETRY_GAP_EPSILON` 虽同为 `1e-9`，却混合了不同职责。G1 至少拆成：

| 语义 | 当前混用点 | G1 要求 |
| --- | --- | --- |
| minimum valid edge length | `EdgeLength` constructor/schema minimum | 产品数据语义；不得由 arithmetic tolerance 顺带决定 |
| edge boundary snap tolerance | progress、remainder、transition/arrival | 由 max ULP、运算链与 `0.01 m` ceiling 推导 |
| minimum geometry extent/nonzero offset | profile/Parking length、width、lateral offset | 产品几何语义；与 overlap tolerance 分离 |
| gap/overlap zero tolerance | leader gap、no-overlap、projection | 由两个 progress 的量化/相减和 hard safety invariant 推导 |
| heading canonicalization tolerance | #123 sampling/normalization | 由 angle ULP 与 `0.0001 rad` ceiling 推导 |

首轮使用 `4 * max ULP` 作为 add/subtract/snap 的保守 representation floor，而不是最终生产常量。在 `16_384 m` progress 上该 floor 为 `0.0078125 m`，低于但已接近 `0.01 m` ceiling；这意味着 raw local `f32` 的可用裕量很窄。候选可通过缩小 guaranteed edge/local range、保留 `f64` authority、使用 residual/compensation 或重新设计 boundary calculation 解决，但必须由 oracle 证明。
