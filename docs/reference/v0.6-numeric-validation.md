## 1. 证据状态

本文是 #122 G1 与 #140 后续复核的可复现研究记录。#122 已完成生产数值面盘点、产品/误差范围、legacy/locality-preserving f64 基线、研究专用 f32/mixed 候选、10k/100k 运行时与内存、production-aligned f64 参照、边界/确定性/长时累计、f16/整数定点矩阵和跨层迁移判断。

2026-07-18 的 #140 进一步证明：#122 的 compensated f32 候选只在累加时维护 residual（补偿残差），却没有在 leader gap、edge remaining、boundary 和 snapshot 中使用它；因此“只有 f64 progress 的 mixed 候选通过”和“f32 没有性能收益”不再是当前有效结论。新的 residual-aware f32 在本研究模型的 10k/100k 矩阵通过现有严格预算，并在同轮 Criterion 中快于 f64。production Core 仍保持 ADR 0012 的现状，直到完整 production 候选、Data/API 迁移和 superseding ADR 完成；研究结果本身不授权静默迁移。

### 2.2 Current range 与 scale fixture

当前 production contract 基本没有物理量上界：`EdgeLength`、`Speed`、`EdgeProgress` 和 profile/Parking fields 接受满足符号约束的有限 `f64`，Core 还有 `f64::MAX` 与累计 overflow 防御测试。首轮候选因此必须分别报告：

- **current compatibility**：对现有无上界 API/schema 的行为变化；
- **product envelope**：项目定位要求正式支持的局部道路范围；
- **pathological range**：极值、overflow 和 rejection seeds；
- **scale fixture range**：为容纳 10k/100k vehicles 而生成的合成 edge 长度。

现有代表性 single-edge builders 的关键范围：

| workload                                     |                          10k edge length | 100k edge length | `f32` ULP at 100k | 说明                                        |
| -------------------------------------------- | ---------------------------------------: | ---------------: | ----------------: | ------------------------------------------- |
| dense / stop-and-go platoon，`6.5 m` spacing |                               `66_000 m` |      `651_000 m` |        `0.0625 m` | 车辆密度有代表性，单 edge 长度是规模夹具    |
| free-flow platoon，`250 m` spacing           |                            `2_501_000 m` |   `25_001_000 m` |             `2 m` | ULP 大于 `10 m/s × 16 ms = 0.16 m` 单步位移 |
| command spatial，`10 m` spacing              |                              `101_000 m` |    `1_001_000 m` |        `0.0625 m` | 主要测 command candidate/query scale        |
| transition-heavy                             | 每条 `5 m`，edge 数随 vehicle count 增长 |             同左 | 小于 `0.000001 m` | locality-preserving，适合 boundary/oracle   |

因此 unchanged builders 继续作为 current compatibility/scale probe；另需增加等 vehicle count、等 density、但每条 edge/local coordinate 不超过 product envelope 的 companion workload。后者只改变 fixture data layout，不实现 #72 的 runtime partition、multi-rate 或并行调度。

### 3.1 Workload 分类规则

每个 `f64`/candidate build 都必须运行相同输入的两组矩阵：

1. **legacy/current-range matrix**：保持现有 builders 不变，用于暴露 range narrowing、stalled progress、ordering 与 overflow 行为；
2. **locality-preserving matrix**：vehicle count、profile、density、step/command sequence 与 observation 数保持一致，仅把合成 mega-edge 分解为 envelope 内的多 edge/多 route fixture，用于评估产品范围内的时间、内存和 cache/SIMD 收益。

不得因为 candidate 在 legacy mega-edge 上失败就直接断言 `f32` 不适合 LaneFlow 产品范围，也不得因为 locality-preserving workload 通过就隐瞒 current public range 的 breaking change。两组结论分别进入 ADR 的 compatibility 与 performance 部分。

### 3.3 f64 locality-preserving 10k baseline（2026-07-18）

测试/benchmark support 在 commit `a2ca90dc60d99f5d7f1e727dc1295ebb263aa217` 增加 companion builders：vehicle count、spacing、profile、初始状态和 60-step sequence 与 legacy builder 相同，只把 route 分成最大 `10_000 m` 的连续 edge。10k free-flow 使用 251 edges，dense/stop-and-go 各使用 7 edges；没有修改 production scalar、control flow、public API 或 schema。

行为对照 `locality_preserving_platoons_match_single_edge_reference_for_sixty_steps` 逐 step 验证 status、route-relative progress、speed、acceleration 和 Vehicle Following safety-projection identity。局部场景产生额外 `VehicleChangedEdge` 是分段布局的预期可观察差异，因此不与 single-edge reference 的总事件数做错误的相等断言；其 edge transition 本身保持连续且所有 edge length `<= 10_000 m`。两条 f64 计算路径的诊断差异处于约 `1e-9` 量级，测试 ceiling 为 `1e-8`，远严于 candidate progress ceiling `0.01 m`。

## 4. 候选实现比较规则

- `f64` 基线必须来自未修改的 production control flow。
- `f32` 或 mixed candidate 必须运行同一 scenario builder、step 数、command 序列和 validation。
- 如使用临时 research-only scalar alias、feature 或完整转换 commit，必须记录 commit/patch；Delivery PR 不得无意保留未接受的 feature 或公共泛型。
- 比较除 wall time 外，还必须覆盖 `VehicleState`、hot scratch buffers、route-distance index、occupancy 和 command spatial index 的 accounted bytes。
- progress candidate 若引入 residual/compensation，residual 本身、padding 和 scratch mirror 必须全部计入 accounted bytes。
- 构造、clone 或 fixture setup 时间与 steady-state step/command 时间分开记录。

## 5. Oracle 与差分合同

同一输入序列下，以 current `f64` 行为作为 reference oracle。候选实现逐 tick 比较：

- tick/time、status、route edge index、event kind/order 等离散结果必须精确一致；
- progress、speed、acceleration、gap 和 constraint distance 按各自领域预算比较；
- `NaN` / Infinity rejection、canonical zero 和错误类别必须一致；
- 失败路径必须保持 world 原子不变；
- boundary 邻域使用 exact boundary、上下一个可表示值和领域 epsilon 两侧的 seeds；
- 覆盖 multi-edge traversal、SignalStop、ParkingStop、RouteEnd、leader change、safe-speed 和 no-overlap projection。

若候选导致离散行为不同，必须记录首次 divergence 的 tick、输入、前后状态和数值差；不得只比较最终位置或平均误差。

### 5.1 Repeated-addition probe

以下 probe 只验证“raw `f32` authority 逐 tick 重复累加”的风险，不替代 production control-flow oracle。每次乘法和加法后都显式 round 到 IEEE binary32；compensated 列使用一个额外 `f32` residual：

| speed / fixed delta / ticks   |   数学位移 |   raw `f32` result |      raw error | compensated result | compensated error |
| ----------------------------- | ---------: | -----------------: | -------------: | -----------------: | ----------------: |
| `10 m/s` / `16 ms` / `6_250`  |  `1_000 m` |    `999.9282837 m` | `-0.0717163 m` |   `1000.0000610 m` |    `+0.0000610 m` |
| `1 m/s` / `16 ms` / `62_500`  |  `1_000 m` |    `999.6366577 m` | `-0.3633423 m` |   `1000.0000610 m` |    `+0.0000610 m` |
| `30 m/s` / `16 ms` / `36_000` | `17_280 m` | `17_288.5664063 m` | `+8.5664063 m` |         `17_280 m` |             `0 m` |

结论仅限于研究路由：静态 ULP 在 local range 内足够，不足以证明 raw repeated addition 满足 long-run progress budget。候选必须比较至少 `raw f32`、`f32 + residual/compensation` 与保留 `f64 progress/integration`；若 residual 使 progress storage 回到 8 bytes，内存结论必须如实体现。

### 6.1 G1 product validation envelope

### 6.2 Candidate acceptance ceiling

候选必须同时满足绝对误差、常规非零值的相对误差和离散行为三类条件：

| 数值域                            |                     最大绝对误差 |                      最大相对误差 | 离散行为要求                                              |
| --------------------------------- | -------------------------------: | --------------------------------: | --------------------------------------------------------- |
| edge length                       |                         `0.01 m` |            `1e-6`，value `>= 1 m` | graph/route validation 分类一致                           |
| edge progress                     | `0.01 m`，每 tick 与 boundary 后 |         `1e-5`，progress `>= 1 m` | transition count/order、remainder、completion 精确一致    |
| bumper gap / route-local distance |                         `0.01 m` |       `1e-4`，`abs(value) >= 1 m` | leader、Passed/Within/BeyondHorizon、projection 精确一致  |
| speed                             |                       `0.01 m/s` |          `1e-4`，speed `>= 1 m/s` | stopped/resume、constraint selection 精确一致             |
| acceleration                      |                     `0.02 m/s^2` |   `1e-3`，`abs(value) >= 1 m/s^2` | comfort/emergency ordering 与 projection attribution 一致 |
| Parking offset/size               |                         `0.01 m` |       `1e-4`，`abs(value) >= 1 m` | bind/arrival/leave 与 validation 分类一致                 |
| local coordinate                  |                         `0.01 m` |       `1e-5`，`abs(value) >= 1 m` | edge binding、sample segment 与 endpoint 结果一致         |
| heading                           |                     `0.0001 rad` | N/A，canonical angle 使用绝对误差 | canonical range、tangent direction 与 wrap 结果一致       |

接近零时不使用相对误差。no-overlap、edge transition、SignalStop、ParkingStop 和 RouteEnd 的离散结果是 hard gate：即使连续误差低于 ceiling，只要这些结果改变，candidate 仍为失败，除非 G1 明确修改行为契约并给出单独迁移依据。

长时段 drift 至少覆盖 60-step representative workload、`36_000` ticks（16 ms 下约 10 分钟）constant-input probe，以及跨多个 edge 后的 residual/remainder 行为；任一连续误差不得无界单调增长。

### 6.3 Epsilon 推导约束

当前 `EDGE_BOUNDARY_EPSILON` 和 `GEOMETRY_GAP_EPSILON` 虽同为 `1e-9`，却混合了不同职责。G1 至少拆成：

| 语义                                   | 当前混用点                                    | G1 要求                                                  |
| -------------------------------------- | --------------------------------------------- | -------------------------------------------------------- |
| minimum valid edge length              | `EdgeLength` constructor/schema minimum       | 产品数据语义；不得由 arithmetic tolerance 顺带决定       |
| edge boundary snap tolerance           | progress、remainder、transition/arrival       | 由 max ULP、运算链与 `0.01 m` ceiling 推导               |
| minimum geometry extent/nonzero offset | profile/Parking length、width、lateral offset | 产品几何语义；与 overlap tolerance 分离                  |
| gap/overlap zero tolerance             | leader gap、no-overlap、projection            | 由两个 progress 的量化/相减和 hard safety invariant 推导 |
| heading canonicalization tolerance     | #123 sampling/normalization                   | 由 angle ULP 与 `0.0001 rad` ceiling 推导                |

首轮使用 `4 * max ULP` 作为 add/subtract/snap 的保守 representation floor，而不是最终生产常量。在 `16_384 m` progress 上该 floor 为 `0.0078125 m`，低于但已接近 `0.01 m` ceiling；这意味着 raw local `f32` 的可用裕量很窄。候选可通过缩小 guaranteed edge/local range、保留 `f64` authority、使用 residual/compensation 或重新设计 boundary calculation 解决，但必须由 oracle 证明。

研究模型直接覆盖 Vehicle Following 数值热链，不声称已经实现一套 f32 Signals/Parking/Data/Spatial production fork。由于唯一满足误差预算的 mixed 候选没有性能收益，ADR 0012 拒绝 production scalar migration；Signals/Parking 等 production f64 行为继续由其现有测试作为 authority。Parking geometry、future local coordinate 与 heading 的 representation floor 由 6.1 和第 7 节量化矩阵覆盖，完整 Spatial oracle 属于 #123。

## 7. `f16` / quantization matrix

Rust 1.96 的 native `f16` 仍不稳定；研究使用 `half 2.7.1` 的 IEEE binary16 conversion。该 crate 为 `MIT OR Apache-2.0`、MSRV 1.81，本轮只加入 dev-dependency；cargo-deny 0.20.2 的 advisories/bans/licenses/sources 全部通过。`half` 文档说明算术可能经 f32 模拟，因此本轮只评价 storage/round-trip，不把它当作 runtime arithmetic 加速证据。

IEEE binary16 在 envelope 内遍历相邻有限值后的 worst half-step：

| Domain          |           Range | binary16 max round-trip error |      Ceiling | Result                                |
| --------------- | --------------: | ----------------------------: | -----------: | ------------------------------------- |
| progress        |  `0..=16_384 m` |                         `4 m` |     `0.01 m` | Fail                                  |
| speed           |   `0..=100 m/s` |                 `0.03125 m/s` |   `0.01 m/s` | Fail                                  |
| acceleration    | `-50..=50 m/s²` |               `0.015625 m/s²` |  `0.02 m/s²` | 单域 Pass，但不足以承担控制 authority |
| extent / offset |  `-128..=128 m` |                   `0.03125 m` |     `0.01 m` | Fail                                  |
| heading         |      `-PI..=PI` |            `0.0009765625 rad` | `0.0001 rad` | Fail                                  |

领域专用整数定点可以满足 round-trip ceiling，但会引入新的 range/saturation/wire contract：

| Domain / format                   |    Representable range |        max error |      Ceiling | Result            |
| --------------------------------- | ---------------------: | ---------------: | -----------: | ----------------- |
| progress `u32` centimeters        |   `0..42,949,672.95 m` |        `0.005 m` |     `0.01 m` | quantization Pass |
| speed `u16` centimeters/s         |        `0..655.35 m/s` |      `0.005 m/s` |   `0.01 m/s` | quantization Pass |
| acceleration `i16` centimeters/s² | `-327.68..327.67 m/s²` |     `0.005 m/s²` |  `0.02 m/s²` | quantization Pass |
| heading `u16` turn                |            `[-PI, PI)` | `0.00004794 rad` | `0.0001 rad` | quantization Pass |

因此 `f16` 禁止用于 Core/Spatial authority；整数定点也不作为 current API/schema 的隐式替换。它们只可在未来的 versioned storage/transport、GPU buffer 或可丢弃 distant presentation cache 中按用途单独设计，必须携带明确 scale/origin、checked overflow、rounding 与 non-authoritative 声明。

### 9.2 被修正的候选语义

旧 `compensated_f32` 只在 `progress += delta` 时保存 Kahan 风格 residual，却在以下位置继续只读取 high component：

- 同 edge 的 leader/follower progress 差；
- 跨 edge 的 remaining distance 与 leader progress；
- edge boundary 判断和归零；
- snapshot 与差分报告。

因此 residual 只是“写时存在、读时丢失”的隐藏状态，不能代表完整的 compensated progress。#140 新增 `residual_aware_f32`，将有效进度定义为 `high - residual`：

```text
same-edge distance
  = (leader_high - follower_high)
  - (leader_residual - follower_residual)

remaining distance
  = (edge_length - progress_high) + progress_residual
```

实现时先把每个 high/residual 分量逐一提升为 f64，再执行减法并转换为 compute scalar，避免 residual 或非近邻操作数先产生不必要的 f32 中间舍入。这不表示所有 high 分量的 f32 减法都会丢位：同号且足够接近的 binary32 操作数在 Sterbenz 条件下可以精确相减；逐分量提升的意义是让实现不依赖这些前提，并让 f64 敏感对照保留完整组合语义。本轮修订前后的已报告误差量级与通过结论一致。`sensitive_f64_control_f32_progress` 使用同一 progress 表示，但把 motion/profile/IIDM/safe-speed/projection compute 与 speed state 升宽为 f64；它用于判断是否真的需要升宽敏感控制链。

最大绝对误差与最大相对误差现在分别保存 tick、vehicle、reference/candidate 值和控制上下文。它修正了此前把两个独立最大值相除、倒推 reference acceleration 的错误做法。具体样本说明根因：

- 100k single-edge dense 的旧 compensated 最大绝对 acceleration 误差发生在 tick 45、vehicle 99998。f64 仍识别 leader，gap 为 `2.645934 m`，acceleration 为 `0.186895 m/s²`；候选的 high progress 量化到 `649_987.4375 m` 后错误失去 leader，acceleration 变为 `1.399999 m/s²`，绝对误差 `1.213104 m/s²`。
- 最大相对误差来自 tick 48 的另一个控制状态：f64 已无 leader、acceleration 为 `1.399998 m/s²`，候选仍保留 leader、acceleration 为 `0.202995 m/s²`，相对误差 `0.855004`。它与上一个最大绝对误差不是同一观测。
- 10 km dense 的旧 compensated 没有丢 leader，但 gap 从 `2.0000955 m` 量化为 `2.0009766 m`。IIDM 的 desired-gap/gap 比值把 `0.000881 m` gap 偏差放大为 `0.002311 m/s²` acceleration 偏差；绝对预算通过，严格相对预算超出约 1.91 倍。
- 10 km stop-and-go 的旧 compensated 最大 `0.501343 m/s²` 误差发生在 hard safety projection：最终速度差约 `0.008021 m/s` 再除以 `0.016 s`，放大为 applied acceleration 差。reference acceleration 约 `-213 m/s²`，所以该样本超绝对预算约 25.1 倍、超相对预算约 2.35 倍，而不是 400–900 倍。
- 旧报告中约 406 倍来自 `8.130737 / 0.02` 的 legacy stop-and-go 绝对预算比；约 855 倍来自 `0.855004 / 0.001` 的另一个 100k dense 相对预算比。它们来自不同场景、不同样本，不能合并解释。

residual-aware f32 修复 gap 读取后，在上述 100k dense 样本把最大 acceleration 绝对误差降到 `0.000007241 m/s²`。因此主要问题是 progress high/residual 的不完整读语义，而不是 IIDM 非线性必须使用 f64。

### 9.8 研究判断与后续治理

1. 建议把 `10_000 m` 作为单 edge 的防御性产品上界候选，而不是日常建模目标长度。edge 应代表拓扑、通行规则、控制和几何连续的有向 lane segment；连接关系、Signal/Parking 控制点、车道属性或几何不连续处本就应拆分。数百公里道路应由 route 组合多个 edge，不应靠一个超长 edge 表示。
2. 10 km 的 binary32 ULP 为 `0.0009765625 m`，四 ULP 为 `0.00390625 m`，低于 `0.01 m` progress ceiling；相比 `16_384 m` 的 `0.0078125 m` 四-ULP floor 留有更合理裕量。4 km/1 km 没有必要仅为数值精度成为 hard cap；100 m 虽在本 workload 更快，却会显著增加 authoring ID、route occurrence 和静态 topology，不应由 microbenchmark 决定。
3. 当前 production API/schema 接近接受全部有限 f64。增加 10 km hard cap 属于行为和 Data validation 变化，不能由 #140 或 #125 静默实施；#126 必须决定 current v0.5 compatibility、下一 format version、loader diagnostic 和迁移策略。
4. residual-aware f32 是值得进入完整 production candidate 的方案；它已经通过当前研究模型的严格连续预算、离散 hard gate、same-runtime replay、长时累计和 10k/100k 性能/内存观察。它仍未覆盖完整 production Signals、Parking、route-distance、command spatial、public error/API/Data normalization，因此不在本轮把 Core 改为 f32。
5. 不建议“敏感控制链遇到误差就升宽到 f64”。当前误差根因已由正确的 high/residual 读取语义解决；额外 f64 control 只提供没有被预算要求的精度，并失去候选内存与吞吐优势。
6. ADR 0012 的 production `f64` 现状继续有效，但其“只有 f64 progress mixed 候选通过、f32 无性能收益”的证据前提已失效。#141 将通过 superseding ADR 比较完整 production residual-aware f32 与现状，而不是在 Accepted ADR 中直接替换结论。
7. #125 的 epsilon 实施应继续等待新的 scalar/edge-range 决策；#126 审计 10 km cap 和 Data/API 迁移；#127 把 edge-cap、argmax provenance、residual-aware oracle 和真实 production 组件纳入长期验证；#128 在新 ADR 与这些实施证据完成后收口。
