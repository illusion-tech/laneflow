## 1. 结论与边界

#110 的 milestone 组合验收已形成可复现候选：

- production current 0.5 canonical Parking + Signals fixture 可经 `laneflow-data` loader 进入 `InitialTrafficData` 与 `CoreWorld`，完成 area-owned 泊位 reserve -> approach/arrival -> commit -> leave -> resume；fixture 同时保留 standalone 路边泊位；

- 六类 Parking command、step composition、failure atomicity、event total order、property/oracle 与 ordered replay 的 D11 矩阵全部有自动化证据；
- 10k static/Reserved matched 三轮、100k matched scaling、command batch、allocation 与 retained-memory 证据齐全；
- stale global max-speed 导致 leave reverse horizon 退化的异常已通过代码复杂度审计、10k/100k matched benchmark 与 Windows CPU profile 三方验证，并以 private on-demand max heap 消除 `batch x V` 重扫；
- shared CI 运行 lifecycle example、workspace tests 和全部 Core benchmark compile，不在 shared runner 使用 wall-clock 数值门禁。

10k / 100% Reserved 三轮中位为 `75.779 ms / 60 ticks = 1.263 ms/tick`：没有达到 `<=1 ms/tick` 方向目标，但低于 `<=4 ms/tick` 硬上限；这是明确保留的非阻断偏差，不写成已达到方向目标。10k matched delta、全部 10k -> 100k ratio 与 retained Parking bytes 均通过目标线。

本切片没有新增或改变 public Core API、data format 或 Adapter API。性能修复只增加 Core private command-spatial 派生 aggregate，不改变 Parking authority、error/event order 或 deterministic result。

## 8. Command batch scaling

三类 workload：

- `reserve/cancel`：100 fixed pairs，200 calls，验证已有 sparse authority path；
- `pathological leave`：依次移除 100 个全局最快 Active vehicles，再执行一次 local leave；
- `six commands`：100 pairs，共 700 Applied calls，覆盖六种 Parking command kind。

命令 workload 使用 `iter_batched_ref`，clone 与 input drop 都在计时区外。审阅中曾发现按值 `iter_batched` 会把 10k/100k `CoreWorld` drop 混入结果；该组旧 artifact 明确作废。six-command 的每个 clone 还在 setup 区执行可逆 reserve/cancel 与 parked-spawn/despawn capacity warm-up，避免 `IndexMap::clone` 收紧 capacity 后，首次 timed spawn 的全 resolver rehash 被误写成 steady command scaling。诊断中该 rehash 在 100k 单独占约 2.42 ms；隔离后消失。Copilot 审阅又发现 `short` route handle 曾在 100-pair 循环内重复解析；最终 helper 将其提升为每批一次，`final3` 三轮在该修复后重新测量。逐阶段审计没有 command x V scan。

## 9. Pathological scaling 的代码审计与 profile

### 9.1 根因与修复复杂度

原实现只保留 successful tick 的 scalar global max speed。若 command phase 移除当前最快 Active vehicle，该值会在下一 tick 前保持保守但 stale；leave reverse horizon 可能从 physical-edge local-k 扩大到整条长边。调查时同一构造在 128/10,000 background 下分别访问 128/10,000 occupants，暴露 global-V 退化；结果语义仍正确，但 fixed command batch 可能形成 `B x V` 工作量。

修复增加 private、按需、可复用 `BinaryHeap<(speed, vehicle)>`：

- 正常 tick 仍只计算 scalar max，并把 heap 标记 invalid；steady step 不构建 heap；
- command phase 首次移除当前 max 时一次 `O(V)` heapify；
- 同 batch 后续移除用 membership handle 做 lazy delete，每项 amortized `O(log V)`；
- batch 总复杂度从最坏 `O(B x V)` 收敛为 `O(V + B log V)`；
- post-removal leave 在 128/10,000 background 下都只检查 1 edge range、访问 0 occupants，heap rebuild count 都恰好为 1。


## 11. Remaining risks

- `<=1 ms/tick` 是方向目标而非当前硬门槛；本机 0% 与 100% Reserved 都略高于该线。后续性能工作应使用受控 power/thermal 环境继续降低 absolute tick cost，不应据此承诺跨机器绝对 SLA。
- first max-speed removal 仍有一次 `O(V)` heapify；这是避免 steady tick heap maintenance 与 `B x V` command 扫描之间的显式权衡。若真实 workload 出现高频“每 tick 只移除一个当前最快 vehicle”，应新建立 profile Issue 比较 always-on indexed max，而不是在 #110 扩张 contract。
- WPR trace 来自 Windows Insider + Balanced power plan，系统 background modules 占据较多采样；结论依赖 operation counter、Criterion scaling 与 profiler 的组合，不把单次 ETL 百分比当作独立门禁。
- 100k 只证明当前 workload 的扩展性；不覆盖 million-scale partition/parallel runtime、cross-platform bit determinism 或 realtime SLA。
- v0.5 仍不包含共享正常行车道停车、free-space maneuver/倒车轨迹、自动选位/排队/收费/充电、Parking Adapter ABI 或 authoring UI。
