## 1. 结论摘要

- 不因开发机稳定版升级到 Rust 1.97.1 而提高最低支持 Rust 版本（MSRV）；工作区继续声明并验证 Rust 1.96。
- 候选 Rust 包全部能在 Rust 1.96 编译。已声明的 MSRV 最高为 `nalgebra` 的 1.89 和 `geo` 的 1.88；`mint` 与 `lyon_geom` 未声明 `rust-version`，只能记录本轮实测兼容。
- `euclid` 是最适合带类型坐标空间的第三方备选；`glam` 是最轻量的游戏数学候选，但缺少点、向量和坐标空间之间的类型隔离。
- v0.6 首版只需要少量三维 `f64` 运算，以及 LaneFlow 专用的校验和错误语义，因此生产实现默认采用 LaneFlow 自有类型，不增加第三方依赖。
- 研究用工作区成员使用 `euclid` 和 `glam` 作为开发依赖，对照当前配置下的内存布局一致性与计算结果；8 个测试在 Rust 1.96.0 和 1.97.1 上均通过。

## 2. 工具链与最低支持版本

MSRV 是“最低支持 Rust 版本”（Minimum Supported Rust Version）的缩写，表示消费者能够构建该包的最低工具链，不等同于开发机当前使用的稳定版。

本机验证：

| 项目                  | 实际值                                |
| --------------------- | ------------------------------------- |
| 最新稳定版            | `rustc 1.97.1 (8bab26f4f 2026-07-14)` |
| 稳定版发布日期        | 2026-07-16                            |
| 最新稳定版 Cargo      | `cargo 1.97.1 (c980f4866 2026-06-30)` |
| 工作区 `rust-version` | `1.96`                                |
| CI 最低工具链         | `1.96.0`                              |
| 宿主目标              | `x86_64-pc-windows-msvc`              |

Rust 官方说明，`package.rust-version` 表示包支持的最低工具链，而不是开发机当前版本；提高它需要权衡维护成本、用户升级成本和新能力收益。Rust 1.97.1 修复了 LLVM 优化阶段的错误编译问题，适合作为当前开发与发布的第二验证轴，但不会自动取消对较低版本的支持。

官方依据：

- [Rust 1.97.1 发布公告](https://blog.rust-lang.org/2026/07/16/Rust-1.97.1/)
- [Cargo `rust-version` 参考](https://doc.rust-lang.org/stable/cargo/reference/rust-version.html)
- [Cargo 对 MSRV 的语义化版本指导](https://doc.rust-lang.org/cargo/reference/semver.html#tooling-and-environment-compatibility)

#123 没有使用仅限 Rust 1.97 的应用程序接口（API），候选依赖也没有要求超过 1.96。因此结论是：

```text
最低支持版本 = Rust 1.96（保持）
开发和最新稳定版验证 = Rust 1.97.1
```

持续集成（CI）当前只执行最低版本的 Rust 检查。是否增加独立的最新稳定版检查任务属于治理与 CI 策略，应单独立项，不应混入 Spatial 权威设计 PR。

### 3.1 最小编译依赖图

审计使用的精确候选配置如下。配置字段属于 Cargo 技术标识符，保持原样：

```toml
euclid = { version = "=0.22.14", default-features = false, features = ["std"] }
glam = { version = "=0.33.2", default-features = false, features = ["std", "f64"] }
nalgebra = { version = "=0.35.0", default-features = false, features = ["std"] }
mint = "=0.5.9"
kurbo = { version = "=0.13.1", default-features = false, features = ["std"] }
lyon_geom = { version = "=1.0.19", default-features = false, features = ["std"] }
geo = { version = "=0.33.1", default-features = false }
rstar = { version = "=0.13.0", default-features = false }
```

聚合临时清单在 Rust 1.96.0 与最新稳定版下都通过 `cargo check --locked`。`cargo tree --edges normal --depth 2` 显示：

- `euclid -> num-traits`；
- `glam`、`mint` 没有常规依赖；
- `kurbo -> arrayvec, polycool, smallvec`；
- `lyon_geom -> arrayvec, euclid, num-traits`；
- `nalgebra` 的 `std` 配置包含 `approx`、`matrixmultiply`、`num-*`、`simba`、`typenum` 等；
- `geo` 即使关闭默认特性仍包含 `geo-types`、`geographiclib`、`overlay`、`rand`、`rstar` 等；
- `rstar -> heapless, num-traits, smallvec`。

这些依赖在本轮显示的许可证表达式都落在 LaneFlow `deny.toml` 允许的 Apache-2.0、MIT、Unicode-3.0 和 Zlib 集合内。生产依赖仍为零；研究原型只有 `euclid` 和 `glam` 两个开发依赖，并由工作区锁文件与 CI 统一审计。

## 4. 独立适配性判断

### 4.1 为什么不直接采用 `euclid`

`euclid` 的幻影单位和空间类型可以在编译期区分标准点与局部点，API 也提供 `Point3D<f64, U>`、`Vector3D`、点积、叉积和归一化。它是最成熟、最接近需求的候选。

但 LaneFlow 公共契约仍必须拥有自己的坐标框架 ID、有限性与带符号零规则、长度绑定、结构化错误、批量失败原子性和序列化语义。即使内部使用 `euclid`，公共边界仍需包装与转换。首版只有数十行点和向量运算，没有矩阵、投影或通用变换图；引入 `euclid` 不会减少最困难的领域代码。因此将其保留为未来内部备选，而不是默认生产依赖。

### 4.2 为什么不因 Bevy 采用 `glam`

`glam` 的 `DVec3` 没有传递依赖，内存布局紧凑，API 直接，Bevy 生态也熟悉它。但 `DVec3` 可以同时被当作点、向量、原点或切向量，不能阻止坐标框架、局部坐标和标准坐标混用。让 Spatial 公共 API 暴露 `glam`，还会把首个适配器的生态选择扩散到 Unity、Unreal 和 Web 消费者。

因此 `glam` 可以在 Bevy 适配器末端消费经过检查的局部位姿，不应成为 Spatial 权威类型。

### 4.3 为什么 LaneFlow 自有类型不是重复实现通用线性代数库

首版自有实现只覆盖：

- 三维 `f64` 点和向量的加、减、点积、长度与归一化；
- 有限性、带符号零和领域构造器；
- 折线线段与累计弧长；
- 长度绑定，以及端点、切向量和上方向采样；
- 经过检查的“局部原点到 `f32`”转换和批量错误。

它不实现矩阵分解、四元数、泛型维度、曲线拟合、地理信息系统、曲面细分或空间索引。只实现 LaneFlow 契约所需的小型原语，范围可以由性质测试和边界测试完整覆盖。

## 5. 容差证据

#122 的局部范围上限 `16_384 m` 处，`f64` 最低有效位单位（ULP）约为 `3.63797880709171e-12 m`。G1 提议如下：

| 语义                        |        提议值 | 与 ULP 或误差预算的关系                             |
| --------------------------- | ------------: | --------------------------------------------------- |
| 最小线段长度                |    `1.0e-4 m` | 约 27,487,791 ULP；属于输入有效尺寸，不是比较容差   |
| 长度绝对容差下限            |    `1.0e-6 m` | 约 274,878 ULP；覆盖十进制和工具舍入，远小于 `1 cm` |
| 16,384 m 处的长度相对容差项 | `1.6384e-5 m` | 约 `0.016384 mm`，远小于 `1 cm`                     |
| 相连端点容差                |    `1.0e-3 m` | `1 mm`，是位置误差上限的 10%                        |
| 上方向投影长度下限          |      `1.0e-6` | 无量纲；拒绝近垂直朝向基放大误差                    |
| 派生朝向基正交归一容差      |     `1.0e-12` | 无量纲；复核 `f64` 单位长度和正交结果               |

解释：

- 最小线段长度与连接容差属于制品或领域质量限制，不能和浮点相等比较混为一个全局 epsilon（误差阈值）。
- 长度容差使用“绝对项 + 相对项”公式；它允许不同生产工具正常的十进制和平方根舍入误差，但拒绝手工维护造成的厘米级漂移。
- 当前原型只为进度端点对照读取 `1.0e-9 m` 的 Core 边界容差；正式 Core 语义由 #125 拆分，Spatial 不复制新的全局常量。

## 6. 原型覆盖

研究用包：`research/issue-123-spatial-prototype`。

覆盖 8 个测试：

1. 拒绝少于两个点、非有限值、零长度线段和近垂直朝向基；
2. 验证起点、内部顶点和终点的位置与切向量规则；
3. 验证 Core 长度与几何长度在容差内可以绑定，超界时拒绝；
4. 验证 Core 终点精确映射到几何终点；
5. 验证标准坐标偏移很大时先减局部原点再转换为 `f32`；
6. 验证先转换为 `f32` 再减原点确实会丢失 `0.125 m`；
7. 验证批量处理中第二条记录失败时，旧输出保持不变；
8. 对照 LaneFlow 自有类型、`euclid` 和 `glam` 的基本 `f64` 向量结果在研究比较容差内一致，并验证当前配置下大小与对齐一致，不把具体字节数冻结为跨目标契约。

命令：

```powershell
cargo +1.96.0 test -p laneflow-spatial-research --locked
cargo +stable test -p laneflow-spatial-research --locked
```

结果：两套工具链均为 `8 passed; 0 failed`，文档测试为 `0 failed`。

原型限制：

- 不是生产 API，不验证线格式模式或场景清单；
- 尚未实现运行时 `frameId` 与原点身份配对，只验证同一坐标框架内的数值转换顺序；
- 没有 1 万或 10 万记录的性能基准、内存分配与内存占用结论；
- 没有接入真实 Core 快照和句柄；
- 没有 Bevy 坐标轴与 `Transform` 冒烟测试；
- 容差仍需 G1 评审，不能直接复制为生产常量。

## 7. G1 接受结果与后续边界

2026-07-18，[#123 G1 设计冻结判断](https://github.com/illusion-tech/laneflow/issues/123#issuecomment-5010087919)已接受：

- 配套空间包，以及使用原始字节 SHA-256 摘要精确配对交通包与空间包的场景清单；
- 米制、右手、Y 轴向上的标准坐标，以及不定义全局前方向；
- 近垂直道路、道路横坡、坐标参考系统和样条曲线不属于首版目标；
- 空间容差、越界、折线顶点切向量和批量失败原子性规则；
- LaneFlow 自有公共类型、首版不增加生产用几何依赖，以及 MSRV 保持 Rust 1.96；
- G1 后按 Spatial 类型、数据制品、绑定采样、批量位姿、验证性能和收口审阅拆分实施 Issue。

本证据包支持 #123 的 G1 设计冻结，但不授权生产实施。当前 Delivery PR 仍必须完成 Copilot 与人工审阅、required checks 和 G3；后续生产 Issue 必须分别完成自身的 G0、G1 与 G2。
