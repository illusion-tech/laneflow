## 1. 结论摘要

- 不因开发机稳定版升级到 Rust 1.97.1 而提高最低支持 Rust 版本（MSRV）；工作区继续声明并验证 Rust 1.96。
- 候选 Rust 包全部能在 Rust 1.96 编译。已声明的 MSRV 最高为 `nalgebra` 的 1.89 和 `geo` 的 1.88；`mint` 与 `lyon_geom` 未声明 `rust-version`，只能记录本轮实测兼容。
- `euclid` 是最适合带类型坐标空间的第三方备选；`glam` 是最轻量的游戏数学候选，但缺少点、向量和坐标空间之间的类型隔离。
- v0.6 首版只需要少量三维 `f32` 运算，以及 LaneFlow 专用的范围、校验和错误语义，因此生产实现默认采用 LaneFlow 自有类型，不增加第三方依赖。
- 研究用工作区成员使用 `euclid` 和 `glam` 作为开发依赖，对照当前配置下的内存布局一致性与计算结果；9 个测试在 Rust 1.96.0 和 1.97.1 上均通过。
- #133 已新增生产 `laneflow-spatial` crate，只依赖 `laneflow-core` path dependency；首轮公共 API 使用每轴 `±16_384 m` 的 LaneFlow-owned canonical `f32` 类型和 opaque immutable registry，没有泄漏研究候选或宿主类型。
- #135 已增加借用型 `SpatialEdgeInput`、`CanonicalPoseF32`、公开原子构造与确定性单边采样，并在量化后 `f32` 点上执行线段、basis、弧长、长度和 5 mm join 校验；没有新增第三方生产依赖。

## 2. 工具链与最低支持版本

MSRV 是“最低支持 Rust 版本”（Minimum Supported Rust Version）的缩写，表示消费者能够构建该包的最低工具链，不等同于开发机当前使用的稳定版。

本机验证：

| 项目                  | 实际值                                |
| --------------------- | ------------------------------------- |
| 最新稳定版            | `rustc 1.97.1 (8bab26f4f 2026-07-14)` |
| 稳定版发布日期        | 2026-07-16                            |
| 最新稳定版 Cargo      | `cargo 1.97.1 (c980f4866 2026-06-30)` |
| 工作区 `rust-version` | `1.96`                                |
| CI 最低工具链         | `1.96.0`                              |
| 宿主目标              | `x86_64-pc-windows-msvc`              |

Rust 官方说明，`package.rust-version` 表示包支持的最低工具链，而不是开发机当前版本；提高它需要权衡维护成本、用户升级成本和新能力收益。Rust 1.97.1 修复了 LLVM 优化阶段的错误编译问题，适合作为当前开发与发布的第二验证轴，但不会自动取消对较低版本的支持。

官方依据：

- [Rust 1.97.1 发布公告](https://blog.rust-lang.org/2026/07/16/Rust-1.97.1/)
- [Cargo `rust-version` 参考](https://doc.rust-lang.org/stable/cargo/reference/rust-version.html)
- [Cargo 对 MSRV 的语义化版本指导](https://doc.rust-lang.org/cargo/reference/semver.html#tooling-and-environment-compatibility)

#123 没有使用仅限 Rust 1.97 的应用程序接口（API），候选依赖也没有要求超过 1.96。因此结论是：

```text
最低支持版本 = Rust 1.96（保持）
开发和最新稳定版验证 = Rust 1.97.1
```

持续集成（CI）当前只执行最低版本的 Rust 检查。是否增加独立的最新稳定版检查任务属于治理与 CI 策略，应单独立项，不应混入 Spatial 权威设计 PR。

## 4. 独立适配性判断

### 4.1 为什么不直接采用 `euclid`

`euclid` 的幻影单位和空间类型可以在编译期区分标准点与局部点，API 也提供 `Point3D<f64, U>`、`Vector3D`、点积、叉积和归一化。它是最成熟、最接近需求的候选。

但 LaneFlow 公共契约仍必须拥有自己的坐标框架 ID、有限性与带符号零规则、长度绑定、结构化错误、批量失败原子性和序列化语义。即使内部使用 `euclid`，公共边界仍需包装与转换。首版只有数十行点和向量运算，没有矩阵、投影或通用变换图；引入 `euclid` 不会减少最困难的领域代码。因此将其保留为未来内部备选，而不是默认生产依赖。

### 4.2 为什么不因 Bevy 采用 `glam`

`glam` 的 `DVec3` 没有传递依赖，内存布局紧凑，API 直接，Bevy 生态也熟悉它。但 `DVec3` 可以同时被当作点、向量、原点或切向量，不能阻止坐标框架、局部坐标和标准坐标混用。让 Spatial 公共 API 暴露 `glam`，还会把首个适配器的生态选择扩散到 Unity、Unreal 和 Web 消费者。

因此 `glam` 可以在 Bevy 适配器末端消费经过检查的局部位姿，不应成为 Spatial 权威类型。

### 4.3 为什么 LaneFlow 自有类型不是重复实现通用线性代数库

生产自有实现只覆盖：

- 三维 `f32` 点和向量的受检加减、缩放与稳健归一化；
- 有限性、带符号零和领域构造器；
- 折线线段与累计弧长；
- 长度绑定，以及端点、切向量和上方向采样；
- 有界 canonical frame 到宿主放置边界和批量错误。

它不实现矩阵分解、四元数、泛型维度、曲线拟合、地理信息系统、曲面细分或空间索引。只实现 LaneFlow 契约所需的小型原语，范围可以由性质测试和边界测试完整覆盖。

## 5. 容差证据

#133 的 canonical 点范围上限 `16_384 m` 处，`f32` ULP 为 `0.001953125 m`，约 `1.953125 mm`。f32 修订 G1 接受如下参数：

| 语义                 |          接受值 | 与 ULP 或误差预算的关系                      |
| -------------------- | --------------: | -------------------------------------------- |
| canonical 点每轴范围 |     `±16_384 m` | 最大 ULP 约 `1.953125 mm`                    |
| 最小线段长度         |         `0.1 m` | 远大于量化间隔；拒绝无展示价值的短噪声线段   |
| 长度绝对容差下限     |        `0.01 m` | 与最终 `1 cm` 位置误差预算一致               |
| 长度相对容差         |          `1e-6` | 10 km 时为 `1 cm`，另加独立 Core 量化余量    |
| 相连端点容差         |       `0.005 m` | `5 mm`，在最终 `1 cm` 位置误差预算内         |
| projected-up 下限    | `0.008_726_535` | `sin(0.5°)` 的冻结 `f32` 值，等号有效        |
| 最终切线角误差       |          `0.5°` | 相对离线 `f64` 参考，不作为 runtime 第二权威 |

解释：

- 最小线段长度与连接容差属于制品质量规则，不能和浮点相等比较混成全局 epsilon。
- 长度容差为 `max(0.01 m, length × 1e-6) + core_quantization_allowance`；几何误差与 Core 长度量化来源分开记录。
- 相连端点与退化线段必须在转换后的 runtime `f32` 坐标上判断。相邻点量化后重合时拒绝，不能用原始高精度值通过校验后再静默合并。
- 当前原型只为进度端点对照读取 `1.0e-9 m` 的 current-f64 Core 边界值；正式 Core owner 已由 #125 拆分，target-f32 由 #127 离线标定，但 #144 形成性能不迁移（no-go）结论后未进入当前生产。Spatial 不复制新的全局常量。

## 6. 原型覆盖

研究用包：`research/issue-123-spatial-prototype`。该原型保留 #123 当时的 `f64 -> local f32` 候选作为历史对照，不是 ADR 0015 的生产接口。

覆盖 9 个测试：

1. 拒绝少于两个点、非有限值、零长度线段和近垂直朝向基；
2. 验证起点、内部顶点和终点的位置与切向量规则；
3. 验证 Core 长度与几何长度在容差内可以绑定，超界时拒绝；
4. 验证 Core 终点精确映射到几何终点；
5. 验证标准坐标偏移很大时先减局部原点再转换为 `f32`；
6. 验证先转换为 `f32` 再减原点确实会丢失 `0.125 m`；
7. 验证批量处理中第二条记录失败时，旧输出保持不变；
8. 验证批量处理全部成功后复用调用方预留容量；
9. 对照 LaneFlow 自有类型、`euclid` 和 `glam` 的基本 `f64` 向量结果在研究比较容差内一致，并验证当前配置下大小与对齐一致，不把具体字节数冻结为跨目标契约。

命令：

```powershell
cargo +1.96.0 test -p laneflow-spatial-research --locked
cargo +stable test -p laneflow-spatial-research --locked
```

结果：两套工具链均为 `9 passed; 0 failed`，文档测试为 `0 failed`。

原型限制：

- 不是生产 API，不验证线格式模式或场景清单；
- 尚未实现运行时 `frameId` 与原点身份配对，只验证同一坐标框架内的数值转换顺序；
- 没有 1 万或 10 万记录的性能基准、内存分配与内存占用结论；
- 没有接入真实 Core 快照和句柄；
- 没有 Bevy 坐标轴与 `Transform` 冒烟测试；
- 容差仍需 G1 评审，不能直接复制为生产常量。

## 7. G1 接受结果与后续边界

2026-07-18，[#123 G1 设计冻结判断](https://github.com/illusion-tech/laneflow/issues/123#issuecomment-5010087919)接受了 Spatial 分层、制品和初始精度候选。2026-07-19，[#133 f32 修订 G1](https://github.com/illusion-tech/laneflow/issues/133#issuecomment-5015720890)与 ADR 0015 取代其中的 canonical `f64` 精度条款：

- 配套空间包，以及使用原始字节 SHA-256 摘要精确配对交通包与空间包的场景清单；
- 米制、右手、Y 轴向上的有界 canonical frame，以及不定义全局前方向；
- 近垂直道路、道路横坡、坐标参考系统和样条曲线不属于首版目标；
- 每轴 `±16_384 m`、5 mm 连接容差、1 cm 最终误差、折线顶点切向量和批量失败原子性规则；
- LaneFlow 自有公共类型、首版不增加生产用几何依赖，以及 MSRV 保持 Rust 1.96；
- G1 后按 Spatial 类型、数据制品、绑定采样、批量位姿、验证性能和收口审阅拆分实施 Issue。

本证据包同时保留历史原型与修订后的生产验证输入。每个生产 Issue 仍必须完成自身 G0-G4；历史原型通过不替代 #133 的代码验证或后续 #137 的性能 Gate。

## 8. #133 生产类型与注册表基线

#133 将 G1 研究结论固化为 `crates/laneflow-spatial` 生产边界：

- `CanonicalFrameId` 使用与 Core external ID 一致的 1–128 字节 ASCII token 语法。
- `CanonicalPoint3F32`、`CanonicalVector3F32` 与 `CanonicalUnitVector3F32` 字段私有，只接受有限输入；构造和受检命名运算统一把 `-0.0` 规范化为 `+0.0`。
- point 每轴使用闭区间 `±16_384 m`，返回 point 的运算重新校验范围；vector 不套用点范围，因此可以表达两个合法端点之间最多 `±32_768 m` 的差。
- 单位方向使用按最大绝对分量缩放的归一化路径，既拒绝零向量，也避免有限大分量直接平方造成中间溢出。
- `SpatialRegistry` 直接以 opaque Core `EdgeHandle` 为 key，按 `LaneGraph::edges()` 顺序提交 dense entries，并使用私有 `HashMap<EdgeHandle, u32>` 查找；不暴露 Core 数值 index、slot 或 hasher。
- crate-private staged builder 在提交前检查 unknown、duplicate、missing edge 与 `u32` 槽位数量；失败不产生部分 registry，且只借用 `LaneGraph`，不修改 Core。
- `SpatialError` 与 `SpatialAxis` 由 LaneFlow 拥有并实现标准 `Display` / `Error`；#133 增加 point range 错误。外部 ID/转换由 #134，Polyline/length/basis/join/sampling 由 #135，frame/origin/batch 由 #136，性能与性质错误边界由 #137 负责。

生产 crate 的单元/公共 API 测试覆盖 frame ID 长度和字符边界、每轴非有限值与范围边界、带符号零、point 结果越界、vector 运算溢出、零方向、极大/极小有限向量归一化、registry 空/完整覆盖、LaneGraph 稳定顺序、unknown/duplicate/missing handle、`u32` 上界判断、失败原子性，以及 Core 不提供 Spatial registry 时仍可独立构造。完整 workspace、MSRV/stable、Clippy、rustdoc、dependency policy 与 Markdown 结果记录在 #133 的 PR/G3 证据中，不由本文替代动态 Gate 状态。

## 9. #135 生产折线绑定与采样验证

#135 在 `laneflow-spatial` 中交付：

- `SpatialRegistry::try_new` 从借用的 `SpatialEdgeInput<'a>` 构造私有 dense 折线条目，输入点只在成功 registry 中保留一份 owned runtime 副本；Data crate 不成为 Spatial 依赖。
- 每条折线至少两个点；每段严格长于 `0.1 m`，basis 的 projected-up 长度必须大于等于 `0.008_726_535`，`f32` 累计弧长必须有限且严格递增。
- 长度绑定采用 `difference <= max(0.01 m, 1e-6 × max(core, geometry)) + 0.0 m`；连接按 LaneGraph/next-edge 稳定顺序检查 runtime `f32` 端点，距离 `<= 0.005 m` 接受。
- current-f64 `EdgeProgress` 使用精确闭区间，不复制 Core 私有 epsilon；零值和末值直接命中端点，内部顶点选择出段，最终端点选择最后入段。
- `CanonicalPoseF32` 只暴露已检查的 point/unit tangent/unit up；unknown/duplicate/missing、几何、长度、join 和进度失败均为携带 edge/index 的 LaneFlow-owned 结构化错误。
- #134 的 `LoadedSpatialPackage` 通过 iterator 直接映射为借用输入。正例使用闭合的 5 m 自连接折线验证构造和内部顶点右连续；现有 canonical fixture 的开放 `loop` 则稳定返回 5 m 自连接断缝错误。

## 10. 后续性能 Gate

#137 在固定参考机器和 release 构建上使用相同输入比较 F32 production candidate 与 F64 reference：

- 10k 位姿提取 p95 `<= 2 ms`，100k p95 `<= 20 ms`；
- 稳态零分配并复用输出缓冲区；10k 到 100k 扩展不超过 `12x`；
- retained memory 至少比 F64 candidate 低 `25%`；F32 throughput 不得慢超过 `5%`；
- 最终位置误差 `<= 1 cm`、切线角误差 `<= 0.5°`，无 NaN/Infinity/部分输出；
- 单独报告 `EdgeHandle -> slot` 与 slot-resolved sampling，高频路径不解析 external ID。
