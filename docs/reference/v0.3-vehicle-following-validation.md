# v0.3 Vehicle Following 验证与性能基线

**文档状态**: Active  
**最后更新**: 2026-07-14  
**适用范围**: v0.3 Vehicle Following 的确定性、不变量、生命周期、10k 性能和 100k 扩展性验证  
**过程证据**: [GitHub Issue #77](https://github.com/illusion-tech/laneflow/issues/77)  
**设计依据**: `../design/vehicle-following.md`、ADR 0003、ADR 0006

## 1. 目的与结论边界

本文保存 [#77](https://github.com/illusion-tech/laneflow/issues/77) 建立、并由 [#86](https://github.com/illusion-tech/laneflow/issues/86) 在最终 candidate-state scratch 复用实现上重新测量的 reference desktop 验证基线，供 M0.3 全面审阅和后续同机性能回归比较使用。GitHub Issue / PR 继续负责 Gate、review thread、Project 状态和合并时序等动态证据。

本基线证明当前实现：

- 在受控测试输入下保持同版本、同环境的状态与事件确定性；
- 覆盖 Vehicle Following 的主要几何、安全、生命周期和失败原子性边界；
- 10k 常规场景达到目标 median `<= 1 ms/tick`，并低于 G3 硬上限 `<= 4 ms/tick`；
- 10k 到 100k dense platoon 未显示全局 `O(V^2)` 增长信号。

本基线不构成跨 CPU bit-level determinism、100k/1M 实时承诺或城市级交通模型验收。

## 2. 行为与不变量覆盖

| 设计要求                                                                  | 主要自动化证据                                                                           |
| ------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| 相同 world/input 逐 tick 状态与事件一致                                   | `tests/determinism.rs`；`vehicle_following_properties.rs`                                |
| 初始 vehicle 输入排列不影响结果                                           | `vehicle_following_properties.rs`；occupancy input-order tests；repeated-edge cycle test |
| same/cross edge、branch、shared edge、repeated edge、self exclusion       | `world::occupancy_tests`；`tests/vehicle_following.rs`                                   |
| overlap rejection、epsilon 接触和 min-gap-only 合法状态                   | `world::occupancy_tests`；lane graph/spawn tests                                         |
| Active/Stopped/Completed occupancy                                        | occupancy status tests；route completion tests                                           |
| IIDM、safe-speed、ballistic 与 route-end cap                              | `longitudinal::tests`；`tests/vehicle_following.rs`                                      |
| acyclic platoon、multiple followers 与 explicit cycle                     | `tests/vehicle_following.rs`；属性测试生成的线性 platoon                                 |
| spawn/despawn、stale handle、slot generation 与失败原子性                 | `tests/handles.rs`；despawn/replacement lifecycle test；route failure tests              |
| candidate state allocation 跨 tick / clone 复用，成功与失败均恢复 scratch | `world::tests::candidate_state_scratch_*`；derived scratch equality test                 |
| projection/transition/completion 事件数量与顺序                           | `tests/vehicle_following.rs`；`tests/route_following.rs`                                 |
| finite、非负速度和 normal-mode no-overlap                                 | 属性测试；10k functional smoke test                                                      |

属性测试使用 `proptest 1.11.0`，其 MSRV 为 Rust 1.85，许可为 `MIT OR Apache-2.0`；它只属于 `laneflow-core` 的 dev-dependencies，不进入 runtime normal dependency tree。新增传递依赖同样使用 permissive license。测试协议为：

- 每次运行 128 个合法线性 platoon case；
- 每个 case 包含 2 到 24 辆车、0 到 20 meter 初始 bumper gap、0 到 30 meter/second 初始速度；
- 每个 case 连续运行 1 到 24 个 100 ms fixed ticks；
- 同时构造正常和反序输入 world，精确比较每 tick 的 `StepResult` 与 `CoreWorld`；
- 每 tick 验证 finite state、非负速度/progress、无物理重叠和每车最多一个 projection event；
- 最小化后的失败 seed 持久化到 `crates/laneflow-core/proptest-regressions/`，并要求随修复提交。

属性测试只生成单 edge 线性 platoon。复杂拓扑和 lifecycle 由可解释的定向回归测试负责，避免用随机生成替代明确边界断言。

## 3. 10k Functional Smoke

常规 CI 运行 `vehicle_following_scale.rs`，对以下四个 10,000 vehicle 场景各执行一个 16 ms tick：

- `free_flow`：大间距、正常巡航；
- `dense_platoon`：2 meter bumper gap 的低速连续 platoon；
- `stop_and_go`：dense platoon 的最前车显式停止；
- `projection_heavy`：5,000 组零 bumper gap follower/leader，预期产生 5,000 个 projection events。

Smoke test 验证执行成功、场景预期事件数量、状态 finite、速度非负和无物理重叠。CI 只使用功能断言，不以共享 runner wall-clock 阻断 PR。

## 4. Reference Desktop 与测量协议

| 项            | 值                                                                           |
| ------------- | ---------------------------------------------------------------------------- |
| CPU           | AMD Ryzen 9 9955HX，16 cores / 32 logical processors                         |
| OS            | Windows 11 Pro Insider Preview 10.0.29617，build 29617                       |
| Rust          | `rustc 1.96.0 (ac68faa20 2026-05-25)`，`x86_64-pc-windows-msvc`，LLVM 22.1.2 |
| Cargo profile | Cargo 默认 optimized bench/release profile                                   |
| 电源模式      | 平衡                                                                         |
| fixed tick    | 16 ms                                                                        |

Criterion 场景构建和每个样本的 world clone 位于 `iter_batched` setup，不计入测量 routine。10k/60-tick 场景使用 20 samples，100k/60-tick 使用 10 samples；warm-up 为 1 second，目标 measurement time 为 5 seconds。Criterion 在单个 routine 较长时会自动延长采样时间以完成样本数，这不改变场景 workload。

三轮命令只改变 baseline 名称：

```powershell
$env:LANEFLOW_BENCH_100K='1'
cargo +1.96.0 bench -p laneflow-core --bench core_step --locked -- --noplot --save-baseline v0.3-scratch-final-round-1
cargo +1.96.0 bench -p laneflow-core --bench core_step --locked -- --noplot --save-baseline v0.3-scratch-final-round-2
cargo +1.96.0 bench -p laneflow-core --bench core_step --locked -- --noplot --save-baseline v0.3-scratch-final-round-3
```

每轮结果从对应 `estimates.json` 的 `median.point_estimate` 提取，不把 Criterion 控制台默认 point estimate 误写为 median。本文基线值是三轮各自 median 的再次取中位数。

## 5. 三轮性能结果

| 场景                      |        Workload | Round 1 median | Round 2 median | Round 3 median |   三轮基线 |   每 tick |
| ------------------------- | --------------: | -------------: | -------------: | -------------: | ---------: | --------: |
| free-flow                 |  10k x 60 ticks |      37.147 ms |      37.943 ms |      37.810 ms |  37.810 ms |  0.630 ms |
| dense-platoon             |  10k x 60 ticks |      55.483 ms |      55.873 ms |      55.768 ms |  55.768 ms |  0.929 ms |
| stop-and-go               |  10k x 60 ticks |      55.219 ms |      55.653 ms |      56.236 ms |  55.653 ms |  0.928 ms |
| projection-heavy          |    10k x 1 tick |       0.956 ms |       0.982 ms |       0.998 ms |   0.982 ms |  0.982 ms |
| transition-heavy          |  10k x 60 ticks |      76.101 ms |      74.816 ms |      72.139 ms |  74.816 ms |  1.247 ms |
| dense-platoon observation | 100k x 60 ticks |     639.609 ms |     641.419 ms |     638.933 ms | 639.609 ms | 10.660 ms |

判断：

- 三个 10k 常规场景均达到 `<= 1 ms/tick` 目标，且明显低于 `<= 4 ms/tick` G3 硬上限。
- projection-heavy 单独报告，不与常规门槛混算；本轮产生 5,000 个 projection events，median 为 `0.982 ms/tick`。
- transition-heavy 是每车每 tick 都发生 self-loop edge transition 的额外压力场景，不属于常规门槛。三轮极差约为三轮基线的 `5.3%`，仍使用三轮中位数而不是单轮结果判断回归。
- 相对 #77 的上一版三轮基线，free-flow、dense-platoon、stop-and-go、projection-heavy 与 100k dense 分别变化 `-1.0%`、`-1.0%`、`-0.7%`、`-1.1%` 和 `-6.0%`；transition-heavy 变化 `+0.8%`。所有变化均低于 `20%` 分析阈值，candidate-state scratch 修复未引入性能回退。
- 100k/10k dense 三轮基线耗时比为 `11.47x`，对应 vehicle 数量增加 `10x`。两个规模点不能证明渐近复杂度，但结果显著低于二次增长应出现的约 `100x`，未发现全局 `O(V^2)` 信号。

基线建立后，同机、同协议三轮基线回退超过 `20%` 必须分析，超过 `30%` 默认阻断，除非记录显式例外。不得把不同 CPU、OS、电源模式或工具链的结果直接套用该阈值。

## 6. 1M 历史观察

[#71 D7](https://github.com/illusion-tech/laneflow/issues/71#issuecomment-4951438925) 保存了 v0.2 steady-state 临时 harness 的 1M 结果：约 `16.39-16.97 ms/tick`。该 harness 不包含 occupancy、IIDM、edge transition 或事件，只是乐观容量参考，不是 v0.3 Vehicle Following 性能数据，也不构成全市实时能力声明。

百万级 partition、parallel、multi-rate、mesoscopic/aggregate 和内存布局由 #72 继续研究，不阻塞 M0.3。

## 8. 剩余边界

- 本基线不测跨 CPU / 浮点实现的 bit-level determinism。
- 10k smoke 在 CI 中只执行一个 tick；连续 60 ticks 由本地 Criterion workload 和定向行为测试覆盖。
- 100k 是扩展性观察，不是高精度实时承诺。
- transition-heavy 的轮间波动高于常规场景，应继续作为非阻断压力指标观察。
- 当前没有 untrusted input fuzzing、signal/intersection/lane-change 或城市级调度验证；它们不属于 #77。

## 9. 收口判断

#77 的行为、不变量、10k 性能和 100k 扩展性验收输入已经形成；#86 进一步证明 candidate state allocation 已移出 tick 路径，且最终实现的同机三轮性能没有回退。本文可以作为 #17 M0.3 全面审阅的独立证据，但不能替代 #17 自身的 G3/G4 和里程碑完成审计。
