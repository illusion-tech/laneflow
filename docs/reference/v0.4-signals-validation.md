# v0.4 Signals 验证与性能基线

**文档状态**: Active  
**最后更新**: 2026-07-15  
**适用范围**: v0.4 Signals 的 loader-to-Core、确定性、SignalStop、10k 性能与 100k 扩展性验证  
**过程证据**: [GitHub Issue #97](https://github.com/illusion-tech/laneflow/issues/97)  
**设计依据**: `../design/signal-system.md`、ADR 0003、ADR 0009

## 1. 结论边界

本文保存 #97 的长期可复用验证协议与 reference desktop 基线。GitHub Issue / PR 继续负责 Gate、review thread、final-head commit、Project 状态和合并时序等动态证据。

本基线证明当前实现：

- canonical v0.4 JSON 经 production loader 后可直接驱动 controlled/none Gate、green/yellow/red、排队与放行；
- 相同 normalized state 和 tick input 在同版本、同环境中逐 tick 重放出完全相同的 CoreWorld、query snapshot 与 ordered events；
- fixed-time resolver 在 1–8 groups/phases、边界、wrap、长时间与 `u64::MAX` 附近匹配独立 `u128` oracle；
- 10k matched workload 覆盖 no-Signals、all-none、all-green、all-red、stop-release 和 mixed-offsets；
- 100k observation 同时把 vehicles 与 signal topology 放大 10 倍，用于排查全表扫描和接近二次增长的信号。

本文不构成跨 CPU bit-level determinism、100k 实时承诺、城市级交通模型验收，也不覆盖 permissive movement、红灯右转、冲突区或 actuated/adaptive controller。

## 2. 行为、属性与失败原子性

| 设计要求 | 自动化证据 |
| --- | --- |
| current fixture 经 production loader 进入 Core | `laneflow-data/tests/signals_end_to_end.rs` |
| controlled/none Gate 与 green/yellow/red query | `signals_end_to_end.rs`、`signals_runtime.rs` |
| 黄/红停车、队列保持、green release、Gate traversal | `signals_end_to_end.rs`、`signals_compliance.rs` |
| 相同输入精确 replay | `signals_determinism.rs` |
| failed step 不改变 authority state，retry 等于 fresh world | `signals_determinism.rs`、`signals_runtime.rs` |
| controller 后 group 的稳定事件总序 | `signals_determinism.rs`、`signals_runtime.rs` |
| 1–8 groups/phases 的 resolver 属性 | `signal::tests::fixed_time_resolver_matches_independent_u128_oracle` |
| exact boundary、wrap、任意长时间、near-overflow | 独立 `u128` oracle 属性测试与定向 overflow test |
| 10k topology、finite state、no-overlap、事件语义 | `signals_scale.rs` |
| 100k matched topology pressure smoke | `signals_scale.rs::mixed_offset_100k_pressure_smoke`（显式 ignored test） |

属性测试由 `proptest 1.11.0` 生成 1–8 个 groups、1–8 个 phases、正 duration、任意 aspect seed、任意 canonical offset 和任意 `u64` time。Oracle 独立使用 `u128` 求 cycle position，再以顺序区间解析 phase；不调用生产 resolver，也不增加任意时间 public API。

端到端测试直接 `include_str!` #94 的两个 canonical fixtures，不复制第三份 JSON。`v0.4-signals-baseline.laneflow.json` 验证完整 Signals；`v0.4-empty-signals.laneflow.json` 的无信号 Core route/repeated-edge 行为继续由 `example_route_data.rs` 验证。规模压力只使用 Rust builder，不伪装成外部数据 fixture。

## 3. Matched workload

所有 Signals 规模场景共享 `tests/support/signals_validation_scenarios.rs`，车辆、route horizon 和 profile 保持一致：

| 规模 | Active vehicles | Controllers | Groups/Gates/Routes | 每 route 车辆 | Tick |
| --- | ---: | ---: | ---: | ---: | --- |
| 10k | 10,000 | 100 | 400 | 25 | 16 ms x 60 |
| 100k | 100,000 | 1,000 | 4,000 | 25 | 16 ms x 60 |

10k 六类输入：

- `no_signals`：相同 graph/routes/vehicles，空 SignalRegistry；
- `all_none`：400 StopLines/Gates，全部 `signalControl:none`；
- `all_green`：100 controllers、400 groups，固定 green；
- `all_red`：固定 red，形成 denied queue；
- `stop_release`：480 ms red + 480 ms green，覆盖同一 60-tick window 内的停车和放行；
- `mixed_offsets`：320 ms green、160 ms yellow、320 ms red、160 ms all-red，controller offset 以 16 ms 递增并 canonical wrap。

常规 CI 对 10k 六类场景运行 60 ticks，验证 matched uncontrolled/green 行为等价、受控模式关键事件、拓扑数量、finite state 与 no-overlap。100k smoke 是显式 G3 命令；常规 CI 只编译 benchmark，不用共享 runner wall-clock 阻断。

## 5. 性能结果

验收阈值固定为：

- 10k common target `<= 1 ms/tick`，hard limit `<= 4 ms/tick`，60 ticks `<= 240 ms`；
- no-Signals 相对同机同次 v0.3 等价场景回归 `>20%` 必须分析，`>30%` 默认阻断；
- matched all-green、all-none 相对 no-Signals 的增量 `>20%` 必须 profile；
- 100k mixed-offset / 10k mixed-offset 目标 `<= 20x`；
- hot-path 审计发现 vehicle x all signals、per-vehicle allocation 或接近二次增长时直接阻断。

| 场景 | Round 1 median | Round 2 median | Round 3 median | 三轮基线 | 每 tick |
| --- | ---: | ---: | ---: | ---: | ---: |
| no-signals 10k x 60 | 58.399 ms | 58.537 ms | 57.359 ms | 58.399 ms | 0.973 ms |
| all-none 10k x 60 | 60.670 ms | 58.437 ms | 58.078 ms | 58.437 ms | 0.974 ms |
| all-green 10k x 60 | 60.228 ms | 58.554 ms | 57.197 ms | 58.554 ms | 0.976 ms |
| all-red 10k x 60 | 70.123 ms | 68.261 ms | 67.453 ms | 68.261 ms | 1.138 ms |
| stop-release 10k x 60 | 63.658 ms | 64.068 ms | 63.168 ms | 63.658 ms | 1.061 ms |
| mixed-offsets 10k x 60 | 67.140 ms | 67.897 ms | 67.135 ms | 67.140 ms | 1.119 ms |
| mixed-offsets 100k x 60 | 760.030 ms | 759.076 ms | 751.022 ms | 759.076 ms | 12.651 ms |

判断：

- no-signals、all-none、all-green 达到 `<= 1 ms/tick` common target；restrictive pressure 场景为 `1.061–1.138 ms/tick`，略高于 target，但全部远低于 `4 ms/tick` / `240 ms per 60 ticks` 硬上限。
- all-none、all-green 相对 no-signals 三轮基线增量分别只有 `+0.066%`、`+0.267%`，不触发 `20%` profile 阈值。
- mixed-offsets 100k/10k 比为 `11.306x`，低于 `20x` 目标；车辆和 signal topology 同时放大 10 倍时未显示接近二次增长。
- 初次预采样的 all-green 为 72.266 ms，相对 no-signals 59.800 ms 增加 `20.8%`。Hot-path 审计发现全 green 时仍对每辆车计算 signal braking horizon 并查找 Gate；candidate 增加 private committed `has_restrictive_group` bit 后，正式三轮增量降至 `0.267%`。该优化不改变 public API、query 或 event contract，并由 matched equivalence、属性和全量测试覆盖。

同一三轮运行也保留了 v0.3 legacy routines：

| Legacy 场景 | Round 1 | Round 2 | Round 3 | 三轮基线 | 相对 v0.3 文档基线 |
| --- | ---: | ---: | ---: | ---: | ---: |
| free-flow 10k x 60 | 42.023 ms | 42.370 ms | 42.079 ms | 42.079 ms | +11.29% |
| dense-platoon 10k x 60 | 65.697 ms | 66.162 ms | 65.742 ms | 65.742 ms | +17.88% |
| stop-and-go 10k x 60 | 65.916 ms | 67.500 ms | 66.781 ms | 66.781 ms | +19.99% |
| projection-heavy 10k x 1 | 1.079 ms | 1.131 ms | 1.099 ms | 1.099 ms | +11.93% |
| transition-heavy 10k x 60 | 93.908 ms | 96.416 ms | 92.778 ms | 93.908 ms | +25.52% |
| dense-platoon 100k x 60 | 762.728 ms | 777.360 ms | 756.702 ms | 762.728 ms | +19.25% |

这些对比使用同一环境但不同日期的 v0.3 文档基线，全部 legacy routine 同向变慢 `11–26%`，而不是 Signals matched 场景的孤立回退。只有非 common、每车每 tick 都发生 self-loop transition 的 `transition-heavy` 超过 `20%` 分析阈值；没有 routine 超过 `30%` 阻断线。#97 没有改动 Vehicle Following 算法，正式同轮 matched all-none/green 又证明新增快路径无额外成本，因此把该差异记录为 reference desktop 跨运行漂移与 v0.4 Core step 累积开销，后续回归继续使用本表作为当前同机基线。

## 6. Hot-path 审计

实现检查以 `signal-system.md` 第 13 节为准。当前结构的预期证据是：

- load/world initialization 预解析 external IDs、MovementGate dense index 和 route transition metadata；
- tick-start Gate query 通过 route-local `next_controlled_transition` 与 dense snapshots 获取，不扫描所有 Gates/Controllers/Groups；
- fixed-time resolver 每 controller 顺序解析自身 phase program，车辆 traversal 只处理实际跨越的 route transitions；
- signal candidate scratch 跨 tick 复用 capacity，正常 vehicle hot path 不做 per-vehicle signal collection allocation；
- committed signal snapshot 维护 private `has_restrictive_group` bit；全 green 或无 controlled groups 时跳过逐车 braking horizon/Gate denial 搜索；
- 100k/10k matched ratio 与 profiler/代码检查共同判断，两个规模点本身不证明渐近复杂度。
