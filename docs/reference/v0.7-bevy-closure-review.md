| 领域           | v0.7 完成后的唯一当前事实                                                                                                                                |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 版本与工具链   | 只支持 Bevy `0.19.x`，lockfile 固定 `0.19.0`；LaneFlow MSRV 保持 Rust `1.96.0`                                                                           |
| 默认依赖面     | `laneflow-bevy` 默认无 feature，只直接依赖 `bevy_app`、`bevy_ecs`、`bevy_time`、`bevy_transform`、`laneflow-core` 与 `laneflow-spatial`                  |

| 调试边界       | `debug-gizmos` 为非默认 opt-in，只消费当前已验证 batch，运行时默认关闭且预算为零；不产生第二套 Spatial authority                                         |
| 示例边界       | `native-example` 为非默认 opt-in，读取并校验仓库 campus 制品；window/render/input/camera/mesh/material 只存在于 example                                  |

## 4. 正确性、确定性与原子性

headless tests 覆盖 0/1/多 fixed step、catch-up、不同 outer-frame 分块、绑定生命周期、稳定部分映射、Y-up/`-Z` forward、frame-root rebase、frame/token/finite/entity 错误顺序与整批失败原子性。campus E2E 走通真实 scenario/traffic/spatial 制品加载、Core step、Spatial batch 与 Bevy Transform 写入。

Presentation 在每个 outer frame 撤销上一帧 validated 标记；只有本帧 extraction、映射、Entity/parent、有限值与 Transform staging 全部成功后，才发布可供 Gizmos 消费的 batch。未绑定记录按稳定顺序跳过，失败时 `applied_records = 0`，不存在部分提交或 stale debug fallback。

## 5. 资源与性能裁决

#171 的机器可读证据以 source commit `d7e8b1e2313fdadb318433acaa6ed4c998683456` 为输入，记录 Rust 1.96 固定 Windows 性能机、命令、五轮样本、后台负载与适用边界。完整 `PostUpdate` 测量包含 Spatial batch extraction、Adapter validation/mapping、local Transform write 与 Bevy propagation，不包含 Core 交通求解和 renderer。

| 规模 |  p95 中位数 | 冻结上限 |  headroom | allocation/reallocation |
| ---: | ----------: | -------: | --------: | ----------------------: |
|  10k |  `3.067 ms` |   `4 ms` | `23.333%` |                 `0 / 0` |
| 100k | `35.852 ms` |  `40 ms` | `10.369%` |                 `0 / 0` |

10k 到 100k 扩展为 `11.691x`，低于 `12x` Gate；两种规模的 allocated bytes 均为零。该固定机结果是版本收口证据，不是跨平台 SLA。原始轮次与断言见 `v0.7-bevy-performance-evidence.json`，协议和范围见 `v0.7-bevy-validation.md`。

## 9. 兼容性与非目标

- 兼容线只承诺 Bevy `0.19.x` 与 Rust `1.96.0`；Bevy 0.20 需要独立迁移 Issue 和完整重审。
- native example 已在 Windows 本机运行；共享 CI 在 Ubuntu 编译并选择 X11。当前证据不构成 macOS、WASM、mobile 或 Wayland 支持承诺。
- v0.7 只支持单 App/单活动 Session/单活动 canonical frame；不支持跨 frame migration。
- Adapter API 在 v1.0 前仍可通过独立 Issue 演进；任何时间权威、mapping 一一性、稳定顺序、placement token、失败原子性或默认依赖面的变化都必须重新进入 G1。
- 第二个 Engine Adapter、foreign-host boundary proof、presentation interpolation、LOD/pooling、prefab/glTF、editor 与 #72 Core simulation fidelity 保持后续独立范围。

## 10. 结论

没有发现阻断 v0.7 整体收口的剩余缺口。#169-#173 已形成最小、可测试、可演示且默认依赖受控的首个 Reference Adapter；性能结论、可视证据与兼容承诺都有明确适用边界。#174 的最终 Delivery PR 通过 required checks、G3 和 main 复验后，可以执行 #174 与父 #121 的 G4，关闭 v0.7 Milestone；后续能力不得借本次收口静默扩展。
