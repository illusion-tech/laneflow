## 1. 背景

ADR 0012 在 #122 阶段选择了生产 Core 连续数值继续使用 `f64`。当时的主要证据是：旧补偿 `f32` 候选超过跟驰和累计误差预算，只有 `f32` 计算配合 `f64` 进度的混合候选通过，并且研究基准没有显示 `f32` 的耗时收益。

#140 证明旧候选并没有实现完整的补偿语义。它只在累加时保存补偿残差（residual），却在前车间距（leader gap）、当前边剩余距离（edge remaining）、边界（boundary）和快照（snapshot）中只读取高位分量（high component）。补全补偿残差感知（residual-aware）读取后，候选在 5 种 edge 布局、3 种 1 万车辆场景和 10 万车辆密集观测中通过原有连续误差预算，并保持前车、约束、投影、跨边和事件等离散行为无分歧。额外把敏感控制链升宽为 `f64` 没有预算必要，还会失去候选的内存和吞吐方向收益。

#140 同时证明，把超长单 edge 显式拆成不超过 10 km 的多个 edge，在当前 1 万/10 万车辆生产 Core 工作负载中没有可归因的稳态、构造或受控跨界性能惩罚。10 km 处二进制 32 位浮点数的末位单位（ULP）为 `0.0009765625 m`，四个 ULP 为 `0.00390625 m`，仍低于 `0.01 m` 的进度连续误差上限。

这些结果推翻了 ADR 0012 的候选唯一性和性能前提，但研究模型仍不能直接授权生产迁移。信号系统、停车系统、路线距离、命令空间索引、公开 Core API 与错误、Data 规范化、完整保留内存和生产工作负载都必须进入同一个迁移与验证边界。

## 2. 决策状态与实施边界

本 ADR 取代 ADR 0012 关于“下一目标数值权威必须继续统一为 `f64`”的决策，但不把尚未发生的代码迁移描述为当前实现：

- 当前生产 Core、Data v0.5 和现有模式（schema）/加载器继续保持 `f64` 与原兼容范围，直到独立的原子迁移议题（Issue）通过自己的 G0-G4 并合入。
- 下一数值契约采用领域化、经过检查的（checked）`f32`、补偿残差感知进度、职责明确的 `f64` 派生值和 Spatial 标准几何；不采用全项目统一标量。
- 本 ADR 接受目标设计，不豁免生产正确性、完整内存、性能、Data 版本和 API 迁移闸口。
- 在生产切换完成前，文档必须同时标出当前实现和已接受目标，禁止让调用方误以为当前 v0.5 已接受新范围或新舍入语义。

## 3. 下一契约的产品范围

合法输入必须落在有明确物理意义和误差保证的产品范围内。Core 领域构造器是最终裁决者，Data 模式/加载器只做同向前置校验；越界时返回结构化错误，不自动拆分、截断或降级精度。

| 数值域                           | 下一契约硬范围                          | 说明                                           |
| -------------------------------- | --------------------------------------- | ---------------------------------------------- |
| 固定步长（fixed tick）           | `1..=1_000 ms`                          | 更长的追赶由调度器拆成多个固定步长             |
| 单 edge 长度                     | `min_edge_length < value <= 10_000 m`   | 最小有效长度由 #125 与比较误差阈值分开冻结     |
| edge 进度                        | `0..=normalized_edge_length`            | 动态权威使用高位/残差，不接受越界进度          |
| 初始、命令和期望速度             | `0..=100 m/s`                           | 约 `360 km/h`                                  |
| Profile 加速度与减速度           | `0 < value <= 50 m/s²`                  | 保留舒适/紧急减速度等跨字段顺序                |
| 期望车头时距（time headway）     | `0 < value <= 60 s`                     | 零值仍是拒绝输入                               |
| 车辆/停车尺寸（extent）          | `0.1..=128 m`                           | 长度、宽度等正尺寸                             |
| 最小间距（min gap）              | `0..=128 m`                             | 与尺寸的最小值语义分开                         |
| edge 局部有符号偏移              | `abs(value) <= 128 m`                   | 不承担世界坐标职责                             |
| 朝向角（heading）                | `[-PI, PI)`                             | 保持标准化角度语义                             |
| route 总长与其他派生值           | 不套用上述输入上界                      | 有限性、溢出检查与查询视距负责约束             |
| Spatial 标准坐标、弧长和标准位姿 | 继续使用 ADR 0013 的 `f64` 标准空间契约 | 不受单 edge 或 Adapter 局部 `f32` 范围反向限制 |

`10_000 m` 是防御性上界，不是日常建模目标。edge 表达拓扑、通行规则、控制和几何连续的有向车道段；连接、车道属性、信号/停车控制点或几何连续性发生变化时本来就应拆分。Core 和 Data 运行时不自动拆 edge，因为那会重写 edge ID、route occurrence（路线中的 edge 出现位置）、控制点绑定、事件归属和空间制品关系；显式拆分属于创作/导入工具的迁移职责。

## 4. 数值权威分层

| 数值职责                           | 已接受的目标表示                           | 规则                                                        |
| ---------------------------------- | ------------------------------------------ | ----------------------------------------------------------- |
| 固定时间增量、时刻、绝对时间与计数 | 经过检查的整数                             | 禁止浮点挂钟时间累计                                        |
| `EdgeLength`                       | 单个经过检查的 `f32`                       | 静态、edge 局部、构造时一次规范化，不使用残差               |
| `EdgeProgress`                     | 私有 `high: f32` + `residual: f32`         | 唯一有效值是逐分量升宽后计算的 `f64(high) - f64(residual)`  |
| `Speed`、Profile、尺寸/偏移        | 经过检查的 `f32`                           | 合法输入受第 3 节产品范围约束                               |
| `Acceleration`                     | 经过检查的有符号 `f32`                     | 表示实际应用加速度；只要求有限且可表示，不套用 Profile 上界 |
| 纵向、占用与约束临时状态           | 以经过检查的 `f32` 为生产候选              | 必须通过完整离散判定基准；不因“敏感控制”普遍升宽            |
| route 距离派生索引                 | `f64` 前缀基线与分块局部 `f32` 候选        | 本 ADR 冻结语义，不预先冻结唯一物理布局                     |
| Data JSON 解析暂存                 | `f64` 或等价的高保真解析值                 | 只用于转换前校验和原始输入诊断，不形成运行时权威            |
| Spatial 标准几何、弧长和标准位姿   | LaneFlow 自有的 `f64`                      | 继续由 Spatial 拥有                                         |
| Adapter 局部位姿                   | 经过检查的局部 `f32` 或显式 `f64` 宿主路径 | 先在 `f64` 中减去局部原点；不得反馈写入 Core                |
| `f16` / 量化整数                   | 仅用于显式编码或缓存                       | 不承担 Core/Spatial 权威                                    |

### 4.1 `EdgeLength` 使用单个 `f32`

`EdgeLength` 是单个 edge 的静态局部长度，不逐 tick 累加。10 km 附近相邻 `f32` 的间隔约为 `0.977 mm`，舍入到最近可表示值（round-to-nearest）的最大一次存储误差约为 `0.488 mm`。因此它不需要残差对，也没有充分的精度理由继续保存为 `f64`。

把字段从 8 字节降到 4 字节，不保证整个 `LaneEdge` 或注册表一定缩小；结构体填充和容器布局必须由完整的内存账本测量。数值表示决策不能冒充已经实现的内存收益。

### 4.2 `EdgeProgress` 使用补偿残差感知表示

`EdgeProgress` 会重复积分，并参与前车间距、edge 剩余距离、边界、Parking 锚点、命令空间索引、快照和跨 edge 余量，因此必须封装高位分量与残差：

- 有效值唯一规定为 `effective = f64(high) - f64(residual)`；任何只读取高位分量的路径都是阻断缺陷。
- 从 `f64` 输入构造时，先经过检查生成高位分量，再保存高位分量与原值之间可表示的残差；拒绝非有限值、分量溢出和有效值越界。
- 标准零值固定为 `+0.0/+0.0`；相等和排序比较有效值，而不是按分量做字典序比较。
- 差值、剩余量、间距、边界和快照先分别把分量升宽到 `f64` 再组合；这只保护补偿语义，不授权把整个控制链重新升宽为 `f64`。
- edge 跨越必须通过领域方法消费规范化的 `EdgeLength`，保留真实余量；只有有效余量命中 #125 冻结的边界吸附规则时才归零并清除残差。
- 高位分量和残差不进入公开字段、JSON、事件或 Adapter API，也不允许被单独序列化。

### 4.3 派生加速度和查询值不受输入上界机械截断

Profile 加速度/减速度的 `50 m/s²` 上界只约束配置输入。硬投影可能产生更大的实际应用加速度；例如 `100 m/s` 在最小 `1 ms` 内降到零时，量级可达 `100_000 m/s²`。这种派生值必须经过检查地计算，并按 `0.02 m/s²` 绝对预算、常规非零值相对预算和离散归因验证，不能截断到 Profile 上界。

制动距离、候选行程、route 查询视距和 route 总长同理：它们必须保持有限且不溢出，不能通过静默饱和掩盖错误。

## 5. 公开 Core API、Data 与诊断边界

公共 API 按真实权威分层，不统一伪装为 `f64`：

- `EdgeLength`、`Speed`、`Acceleration`、Profile 标量、尺寸/偏移等单一 `f32` 数值域的主要构造器和 `value()` 使用 `f32`；其中 `Acceleration` 表示实际应用的有符号派生值，不套用 Profile 配置上界。
- Data、Spatial 或现有 `f64` 生产者通过 `TryFrom<f64>` 或命名明确的受检转换进入单一 `f32` 数值域；禁止调用方先使用未检查的 `as f32`。
- `EdgeProgress` 的构造器和有效值观察接口使用 `f64`，因为把高位分量和残差组合回 `f32` 会再次丢失残差提供的有效精度。
- 实际采用 `f64` 的 route 派生距离与 Spatial 标准值继续暴露 `f64`；Adapter 局部值按输出路径暴露经过检查的 `f32` 或 `f64`。
- 转换错误可以保留被拒绝的原始 `f64` 输入；规范化后的运行时错误使用真实数值域或有效值。错误载荷不统一强制升宽。

Data 的 JSON 词法类型仍是 `number`，但接受范围、舍入、Rust 公共标量、诊断和规范化值都会改变，因此必须按 ADR 0008 原子升级下一有效格式。迁移要求：

1. 模式范围、私有线格式 DTO、加载器路径诊断、固定样例和 Core 构造器在同一迁移切片中更新；
2. 迁移前当前 v0.5 及其生产加载器保持不变；
3. 切换后有效代码树只保留新的当前加载器，不在运行时叠加 v0.5 兼容分支；
4. 真实资产如需转换，使用显式离线迁移工具，不自动拆 edge 或截断；
5. 批量规范化继续使用“先计算、后提交”，任一转换失败不得留下部分 world/data 状态。

## 6. route 距离冻结语义，不预先冻结唯一布局

当前 Core 不是只保存一个冷的 route 总长，而是按 route 出现位置保留前缀偏移、分段总长和终点距离，并用于 `ParkingStop`、`RouteEnd`、跨 edge 命令重叠等查询；Signals 也维护同类“下一受控跨越”距离元数据。因此，route 距离是由同一路线车辆共享、随出现位置增长、部分进入单步执行路径的常驻派生索引。

单 edge 的 10 km 上界不能证明不断增长的裸 `f32` 前缀足够。`f32` 在 100 km、1,000 km 和 10,000 km 附近的 ULP 分别约为 `7.8125 mm`、`62.5 mm` 和 `1 m`；两个巨大前缀相减无法稳定恢复厘米级局部距离。

本 ADR 冻结以下语义：

- 所有 route 距离严格派生自规范化的 Core `EdgeLength`，不保留原始输入的第二份长度权威。
- 有限视距查询、`ParkingStop`、`RouteEnd`、`SignalStop` 和命令重叠必须满足连续误差预算和离散零分歧。
- 当前要求为 O(1) 的查询继续保持 O(1)，或具有不依赖整条 route 长度的明确界限；禁止退回逐 tick 扫描整条 route。
- 任意长的有限 edge 序列继续使用分段且防溢出的语义，不新增“整条 route 总长必须有限”的合法性条件。
- #127 的扩展常驻内存账本必须覆盖 route 距离、Signals route 元数据、结构体填充和 route/车辆比例差异。

#127 至少比较：

1. **`f64` 前缀基线**：每个规范化的 `f32 EdgeLength` 精确提升为 `f64` 后累计，优先简单性和局部差值稳定性；
2. **分块局部 `f32` 索引**：每个出现位置保存有界分块内的 `f32` 局部偏移，分块保存 `f64` 基值/总长，查询在 `f64` 中组合局部尾段与分块差值。

若完整内存账本证明 route 距离占比很小，或分块候选没有可重复收益，应保留 `f64` 前缀基线；理由是收益不足以支付复杂度，而不是把随出现位置线性增长的索引误称为一个冷 `f64`。

## 7. Spatial 标准精度与长度绑定

ADR 0013 的 Spatial 架构、标准 `f64` 坐标、弧长和位姿继续有效。Core 有效进度进入 Spatial 时以 `f64` 观察值提供；这不会把高位分量和残差暴露给 Spatial。

`EdgeLength` 改为规范化的 `f32` 后，长度绑定容差必须显式分离几何计算误差和 Core 长度量化误差：

```text
geometry_tolerance = max(
  SPATIAL_LENGTH_ABS_TOLERANCE_METERS,
  SPATIAL_LENGTH_REL_TOLERANCE * max(abs(core_length), abs(geometry_arc_length))
)

tolerance = geometry_tolerance + core_edge_length_quantization_allowance
```

量化余量必须覆盖合法范围内 `f64 -> f32` 舍入到最近可表示值的最坏误差；精确公式和含等号的边界由 #125 的判定基准冻结，可使用一个完整的局部 ULP 作为保守候选。10 km 处一个 ULP 约为 `0.977 mm`。不能因为旧 Spatial 容差小于该量化误差就让 `EdgeLength` 回到 `f64`，也不能把几何误差和量化误差隐藏在一个来源不明的全局误差阈值中。

容差内仍使用 `ratio = effective_core_progress / normalized_core_length`，再计算 `geometry_s = ratio * geometry_arc_length`，保证 Core 终点映射到几何终点。Adapter 不得用引擎样条长度覆盖 Core 长度，也不得把采样位姿反写为进度。

## 8. 确定性、误差阈值与失败原子性

- 相同 Core 版本、运行环境、初始状态和输入序列继续产生一致结果；跨 CPU、语言或编译器的位级浮点确定性仍不承诺。
- 状态、前车、约束、投影归因、跨 edge、事件种类/顺序和确定性重放是硬性闸口；连续值误差通过不能覆盖离散分歧。
- 非有限值、超产品范围、转换溢出、有符号零、批量命令和加载失败必须在进入权威状态前处理，并保持“先计算、后提交”。
- 当前 `EDGE_BOUNDARY_EPSILON` 和 `GEOMETRY_GAP_EPSILON` 不得直接转换为 `f32` 复用。#125 分别冻结最小有效尺寸、edge 边界吸附、间距/重叠零值、Parking 尺寸和 Spatial 朝向/基向量容差。
- `4 * ULP` 只可作为研究下限。10 km 处四 ULP 为 `0.00390625 m`，低于 `0.01 m` 的进度上限，但生产误差阈值必须由实际运算链和边界判定基准确定。

## 9. 生产迁移闸口

生产迁移必须同时满足正确性和实质收益，不能用仅供研究的标量布局或单轮微基准代替。

### 9.1 正确性闸口

- 现有全部连续误差预算通过；正常非零值继续满足验证文档的相对误差预算。
- 状态、前车、约束、投影归因、跨 edge、事件顺序、确定性重放与失败原子性零分歧。
- 覆盖精确边界、相邻可表示值、单 tick 多 edge、36,000 tick、多轮 route、残差消费/归零、前车变化、`SignalStop`、`ParkingStop`、`RouteEnd`、无重叠、非有限值、超范围、转换溢出和批量/加载失败。
- 当前 `f64`、补偿残差感知的 `f32` 与“`f32` 计算 + `f64` 进度”使用同一生产验证矩阵；旧的仅高位补偿候选只保留为历史回归项。

### 9.2 完整常驻内存闸口

预热后的 1 万和 10 万车辆扩展常驻内存账本，均必须相对当前 `f64` 至少降低 `10%`。账本至少覆盖：

- 车辆槽位和候选状态镜像；
- 纵向、占用和命令空间临时状态；
- route 距离、Signal/Parking 常驻状态；
- 容器容量、结构体填充和 route/车辆比例。

两个 `f32` 分量组成的进度仍占 8 字节；不能宣称进度本身已经减半。收益必须来自完整的生产布局。

### 9.3 性能闸口

- 必测的生产 1 万/10 万车辆工作负载不得出现可重复且无法解释的超过 `5%` 中位数回退。
- 构造与稳态分开测量；固定机器、电源模式、编译器和输入，记录中位数与区间。
- 区间重叠或小于噪声的差异必须复跑，不能挑选单轮最有利数字。
- 补偿残差感知的 `f32` 必须继续与“`f32` 计算 + `f64` 进度”比较；若正确性和完整内存相同，而残差复杂度没有可重复收益，保留 `f64` 进度是有效回退。

### 9.4 回退规则

任一正确性或实质收益闸口失败时，生产回退为保留当前 `f64`。禁止通过放宽误差预算、保留两套构建期精度、值域外 `f64` 回退，或让 `main` 长期处于半迁移状态来强行通过。

## 10. 后果

正向后果：

- 单 edge、车辆热状态和控制参数获得闭合的产品范围，合法输入与误差保证一致。
- 补偿残差感知进度修复旧候选“写时保存、读时丢失”的根因，不需要普遍的敏感控制 `f64` 回退。
- 公共 API 反映真实权威，避免单一 `f32` 值通过 `f64` 取值器假装保留双精度。
- route 距离和 Spatial 保留职责明确的 `f64` 角色，同时允许以完整证据优化随出现位置线性增长的派生索引。
- 迁移必须证明整体收益，不因单字段或研究模型的局部数字承担跨层破坏性变更。

成本与风险：

- Core API、Data 格式、模式范围、错误语义与 Spatial 长度绑定都会改变，需要原子迁移和显式资产转换。
- 高位/残差封装、route 距离候选和领域误差阈值增加实现与审阅复杂度；任何只读取高位分量的路径都可能重新引入离散错误。
- 10 km edge 上界要求创作/导入工具显式拆分旧超长 edge，并维护引用、控制点和空间制品关系。
- `f32` 不保证所有 CPU 上更快；若完整内存或性能收益不足，生产仍可能保留 `f64`。

## 11. 被拒绝的替代方案

### 11.1 所有 Core 数值继续统一 `f64`

它仍是当前生产实现和迁移失败时的可靠回退，但不再是已接受的目标。#140 已推翻“只有 `f64` 进度混合方案通过”和“`f32` 没有性能收益”的前提；不能把尚未完成生产转换当作永久拒绝补偿残差感知 `f32` 的证据。

### 11.2 全量直接使用 `f32`

未补偿的重复进度加法已在长时和控制矩阵中失败；没有残差的全量替换继续拒绝。

### 11.3 遇到敏感控制就升宽为 `f64`

#140 显示主要误差来自丢失残差后的错误间距、前车与投影，而不是 IIDM 必须使用 `f64`。该方案会失去候选的内存与吞吐收益，也没有误差预算要求，予以拒绝。

### 11.4 范围外自动回退到 `f64` 或提供构建期精度开关

它们会形成两套数据布局、确定性路径和测试矩阵，使相同 API 的交通语义随输入或构建参数变化，予以拒绝。

### 11.5 所有公开值继续返回 `f64`

这只能减少部分源码迁移，却会掩盖单一 `f32` 权威的舍入事实。只有补偿进度有效值、实际采用 `f64` 的派生距离和 Spatial 标准值保留 `f64` 表面。
