## 背景

LaneFlow Core 当前拥有车道拓扑、`EdgeLength`、路线区段和边内 `EdgeProgress`，但不拥有把进度映射为世界位置与朝向的道路中心线。若每个引擎适配器（Engine Adapter）独立维护道路样条曲线、长度和采样规则，同一份交通数据会在不同宿主中产生不同位姿，停车系统的边相对几何也没有共同的解析基准。

本 ADR 最初基于 ADR 0012 的当前 `f64` Core 数值边界，确定几何权威属于哪一层、Core 长度与几何弧长如何绑定，以及适配器消费什么稳定输入。ADR 0014 后续接受目标 `f32 EdgeLength` 与补偿残差感知进度；ADR 0015 又把 Spatial runtime 精度修订为有界 `f32` canonical frame。变化不影响本 ADR 的分层、长度权威、制品配对与失败原子性。生产 Core/Data 仍按其独立迁移 Gate 推进。

本文中的“权威职责（authority）”表示某项状态由哪一层定义并最终裁决；“标准坐标（canonical）”表示 LaneFlow 统一使用、尚未转换为宿主坐标的空间；“制品（artifact）”表示可独立版本化和校验的数据文件。组件名 Core、Data、Spatial、Adapter 和 Presentation 分别表示核心层、数据层、空间层、适配层和表现层。

## 决策

### 1. 增加引擎无关的空间层

LaneFlow 增加独立于 Core 和引擎适配器的空间层：

```text
交通数据 -----> LaneFlow Core
   |                 |
   v                 v
空间数据 -----> LaneFlow Spatial -----> 引擎适配器
```

- Core 继续拥有拓扑、边长、路线、进度和交通行为的权威职责。
- Spatial 拥有标准中心线、累计弧长、边绑定和位姿采样的权威职责。
- Core 不依赖 Spatial；未来 `laneflow-spatial` 可以依赖 `laneflow-core` 的不透明句柄和只读输入。
- 适配器消费 Core 快照与 Spatial 位姿，不拥有或复制权威道路几何。
- 网格、材质、地形、路口表面、引擎样条曲线和表现层细节层次（LOD）仍属于适配层与表现层。

### 2. Canonical frame 采用米制、右手、Y 轴向上的有界局部欧氏坐标

- 坐标单位固定为米，角度固定为弧度，标量使用 LaneFlow 自有的 `f32` 类型；点每轴闭区间为 `±16_384 m`。
- 标准坐标系为右手坐标系，`+Y` 表示上方向，`X/Z` 构成水平面。
- 道路前方向由中心线切向量决定，不定义容易被宿主误读的全局 `forwardAxis`。
- 标准坐标框架必须有稳定的 `frameId`；不同框架的点、原点和批次不得混用。
- 地理配准和坐标参考系统（CRS）不进入首版运行时契约，也不使用语义未定义的不透明扩展字段。未来需要地理坐标时，通过独立 ADR 和数据格式版本加入。
- frame 到宿主场景的放置由适配器或调用方选择；它不进入 Core 的车辆高频状态，也不成为几何权威。

### 3. 交通与空间使用配套制品，由场景清单精确配对

- 当前唯一有效的 Traffic Data v0.5 数据包保持不变，不把世界几何加入现有数据包，也不因 #123 提升 `formatVersion`。
- Spatial 中心线使用独立且可版本化的配套空间包。
- 场景清单（scenario manifest）通过制品的统一资源标识符（URI）或其他稳定标识符，以及对原始制品字节计算的 SHA-256 摘要，精确配对交通包和空间包；不得只按文件名、目录邻接或“相同版本号”猜测配对。
- 两个数据包使用相同的交通边外部 ID 完成语义绑定；运行时规范化后再转换为不透明句柄或索引。
- Core、Spatial 和 Data 加载器只接收调用方提供的内存数据，不在运行时隐式读取文件或联网获取制品。
- 空间包与清单的精确 JSON 结构、模式标识符和当前版本策略，由 G1 后的数据规范 Issue 原子交付。

### 4. 首版中心线使用三维折线

- 每条交通边对应一条有向三维折线（polyline），顶点顺序必须与边的行驶方向一致。
- 累计弧长由加载器或绑定器从受检的 runtime `f32` 顶点计算，不作为可以独立漂移的第二份序列化权威数据；派生布局由 #135 冻结。
- 首版不支持贝塞尔曲线、非均匀有理 B 样条（NURBS）、引擎样条曲线、道路倾斜或横坡，以及换道轨迹。未来曲线必须先离线展平为满足误差契约的折线，或通过取代本 ADR 的新决策扩展。
- 相邻退化顶点、非有限坐标、因道路近垂直而退化的 Y 轴向上朝向基、缺失或重复的边绑定，以及超出容差的连接端点，均在绑定前以结构化阻断错误拒绝。

### 5. Core 长度是交通权威，几何弧长是空间权威

- 创作和导出流水线应从同一中心线来源派生几何弧长与 Traffic `LaneEdge.length`，不鼓励手工维护两份长度。
- 绑定时重新计算几何弧长，并用几何绝对/相对容差加 Core `EdgeLength` 量化余量和规范化的 Core 边长比较；超出容差时返回阻断错误，禁止静默修复。当前 `f64 EdgeLength` 的量化余量为零，目标 `f32 EdgeLength` 的余量由 #127 的 ULP/边界离线判定基准冻结；#144 形成不迁移（no-go）结论后没有进入当前生产。
- 容差内的采样使用 `ratio = effective_core_progress / normalized_core_length`，再计算 `geometry_s = ratio * geometry_arc_length`。这样既保证 Core 终点精确映射到几何终点，也把允许差异限制在已验证的绑定容差内；Spatial 不读取进度的高位/残差分量。
- 适配器不得用引擎样条曲线长度覆盖 Core 长度，也不得把位姿反写为新的进度。

### 6. 采样与批量提取使用 LaneFlow 自有类型

- 单边采样语义为“边句柄 + 已验证进度 → 位置 + 切向量 + 上方向向量”。
- 折线内部顶点采用右连续切向量：精确落在顶点时选择出段；最终端点使用最后一个入段。
- 上方向向量由标准 `+Y` 在切向量正交平面上的归一化投影得到。首版拒绝近垂直线段，不表达道路倾斜。
- 超出 Core 进度边界的输入返回结构化错误；只允许在 Core 领域边界容差内吸附到端点，不作一般性的静默截断。
- 批量输入采用已提交 Core 快照的稳定顺序；输出保持相同顺序并携带稳定车辆标识。
- LaneFlow 定义带 frame identity 的 canonical `f32` 输出，不维护默认 `f64` 位姿作为第二份 runtime 权威。批次遵循“全部计算成功后再提交”；任一记录失败时，不得部分修改调用方输出。
- 引擎适配器只把 LaneFlow 位姿映射为 Bevy、Unity、Unreal、Godot 或 Web 的宿主变换；第三方或宿主向量类型不进入 Spatial、Core 或 Data 的公共应用程序接口（API）。

### 7. 首版不增加第三方生产几何依赖

调研确认，`euclid` 最接近带类型坐标空间的需求，`glam` 提供轻量的 `DVec3`。二者都能在 Rust 1.96 上工作。但首版只需要点与向量、有限性校验、累计折线弧长、线性插值和朝向基构造；LaneFlow 自有类型已经承担公共语义与错误边界。

因此首版 Spatial 生产实现默认不增加第三方几何或数学依赖。`euclid` 与 `glam` 只保留在研究原型的开发依赖中，用于布局和结果对照。若后续变换、曲线或类型互操作需求证明自有实现成本更高，再由实施 Issue 按依赖政策重新审计内部依赖。第三方类型始终不得成为公共契约。

### 8. 最低支持 Rust 版本（MSRV）保持 1.96，最新稳定版作为第二验证轴

开发机升级到 Rust 1.97.1 不改变消费者的最低兼容需求。候选 Rust 包已声明的 MSRV 最高为 1.89；研究原型已分别在 1.96.0 与 1.97.1 上通过；#123 不使用仅限 1.97 的语言、标准库或 Cargo 能力。

因此本 ADR 不提高工作区的 `rust-version = "1.96"`。后续应同时验证最低支持 Rust 版本（MSRV）和最新稳定版。若未来真实依赖或实现必须使用更新能力，应通过独立治理判断显式提高 MSRV，而不是跟随开发机静默变化。

## 后果

正向后果：

- 所有适配器共享同一套几何、长度校验和采样语义。
- Core 保持轻量，可以在没有图形宿主的环境中运行，空间几何不会进入交通高频状态。
- 配套制品可被多个适配器和创作工具复用，同时不破坏当前 Traffic Data v0.5。
- 端点、停车位姿和 frame 到宿主的映射获得可测试的共同事实源。
- 首版不增加第三方生产依赖，许可证、编译时间和公共 API 漂移风险最小。

成本与风险：

- 需要新增 Spatial Rust 包、空间数据包及其模式、场景清单和绑定加载器。
- 两个制品必须通过原始字节 SHA-256 摘要和标识符配对，并在加载时完成跨数据包校验。
- 折线切向量在折点处不连续；视觉平滑、网格、样条曲线和道路倾斜仍由后续适配器或创作层解决。
- Y 轴向上的首版设计拒绝近垂直道路；若真实场景需要垂直轨迹或道路倾斜，需要扩展上方向或道路剖面契约。
- Core 长度与几何长度即使在容差内仍可能存在比例差；目标 `f32 EdgeLength` 还会增加显式量化余量，必须由验证证明组合误差不超过位置预算。

## 替代方案

### 让每个适配器自行维护几何

这会复制制品、长度和切向量/上方向语义，并导致同一 Core 进度在不同宿主中产生不同位姿，因此拒绝。

### 把中心线直接放进 Core

这会让 Core 承担世界几何、坐标框架和表现依赖，扩大高频状态与加载边界，并破坏无图形宿主交通层的职责，因此拒绝。

### 把几何合并进当前交通数据包

这会迫使无图形宿主的 Core 消费者携带无关空间数据，并直接改变当前 v0.5 模式。配套制品加场景清单可以保持独立演进，因此拒绝合并。

### 使用引擎样条曲线作为长度权威

不同宿主的样条曲线算法、控制点和浮点实现不一致，无法形成引擎无关的权威，也会使 Core 行为依赖适配器，因此拒绝。

### 首版直接采用 `euclid`、`glam`、`nalgebra`、`geo` 或曲线库

`euclid` 是可接受的内部备选，但当前自有类型已覆盖最小操作；`glam` 不提供带类型的坐标空间；`nalgebra`、`geo`、`kurbo`/`lyon` 和 `rstar` 的能力面分别超出简单三维折线、属于首版非目标或属于 #72 范围。没有足够收益支持增加生产依赖，因此暂不采用。

## 实施与复核

- #123：完成本 ADR、详细设计、Rust 包调研、研究原型与 G1 评审。
- G1 后拆分：Spatial 自有类型与注册表、空间数据与模式及清单、绑定与采样、批量提取、验证与性能，以及收口审阅。
- #125：保持 current-f64 值和行为不变，拆分 Core 的边界、最小长度等数值职责；Spatial 不把研究原型中当前使用的 Core 容差固化为新的全局 epsilon（误差阈值）。
- #127/#144：前者离线标定目标 `f32 EdgeLength` 的 Spatial 量化余量，后者曾随 Core/Data 原子候选启用但因性能不迁移（no-go）结论回退；未来重启需新建议题。
- #133/#134/#135：依次交付 LaneFlow-owned F32 primitive 与 registry 边界、配套制品加载，以及量化后折线/长度/basis/join 校验和确定性单边采样；#135 不复制 current-f64 Core 的私有边界 epsilon。
- v0.7：Bevy Adapter 消费批量位姿，并验证 frame 放置、坐标轴与手性和真实 `Transform` 同步。
- 只有真实曲线、道路倾斜、坐标参考系统、空间索引或宿主类型互操作需求出现时，才重新评估相应 Rust 包与格式扩展。
