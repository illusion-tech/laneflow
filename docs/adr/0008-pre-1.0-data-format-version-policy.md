# 0008 Pre-1.0 Data Format Version Policy

**状态**: Accepted  
**日期**: 2026-07-13  
**适用范围**: LaneFlow 1.0 前的数据格式版本、production loader、schema/fixture 与迁移兼容策略  
**后续应用**: #94 于 2026-07-15 按本政策将唯一 active format 从 v0.3 直接替换为 v0.4；v0.3 历史由 Git 与 closure review 保存

## 背景

LaneFlow 尚未发布 1.0，也没有已确认的外部数据兼容承诺。v0.2 已完成 lane graph 与 route 的设计、实现和收口；v0.3 在此基础上新增 Vehicle Profile、时间单位和后续 Vehicle Following 输入。

此前 ADR 0006、ADR 0007 和 data loading 设计选择保留 v0.2 schema，并让 production loader 同时支持 v0.2/v0.3。这会要求主线长期维护旧 wire DTO、normalization 分支、public version variant、fixture 和交叉版本测试。当前没有实际用户资产或发布承诺为这些成本提供收益，双版本实现反而会使每次活跃格式调整都累积历史分支。

`formatVersion` 仍有独立价值：它能阻止旧资产被当前字段语义误读，并对缺失、过期或未来格式给出明确诊断。保留版本闸口不等于承诺向后兼容。

## 决策

### 1. `main` 只维护一个当前有效的数据格式

在 1.0 前，仓库当前树只维护一个 active data format。#73 完成时的版本为 `formatVersion: "0.3"`；#94 已按本政策切换为 `formatVersion: "0.4"`。

当 pre-1.0 开发需要破坏性修改外部格式时：

1. 提升 `formatVersion`；
2. 替换当前 schema、fixture、private wire DTO 和 loader；
3. 同步更新 active data-format/design 文档；
4. 明确拒绝所有其他版本。

不得仅为保存历史而在 production loader 中累积旧版本分支。

### 2. 版本闸口必填，但只接受当前版本

production loader 先解析最小 version header，再按当前 private DTO 严格解析完整数据：

```text
version header -> current version check -> current wire shape -> Core normalization
```

- 缺失或类型错误的 `formatVersion` 返回结构化 JSON shape error。
- 旧版本和未来版本返回结构化 `UnsupportedFormatVersion`，并包含 expected/actual。
- 对不支持版本，不进入当前 units、unknown field 或 domain validation，避免用当前规则误诊旧/未来格式。
- public loader 返回单一当前 `LoadedPackage`，不公开历史版本 enum/variant。

### 3. Current contract 不激活旧 schema、fixture 或 loader tests

旧 fixture、loader branch 和行为测试仍由 Git history、里程碑收口文档及对应 Issue/PR 保存。active 示例和行为回归必须迁移到 current fixture，不能为了测试历史格式而保留 production compatibility path。

ADR 0011 额外要求：一旦 versioned schema `$id` 进入 public publication catalog，该 schema 文件可以且必须作为 immutable publication artifact 保留，以维持原始 retrieval URL。历史 artifact 不属于 active data format，不由 production loader 接受，不进入 current fixture/normalization matrix，也不得原地修改。

历史文档引用实现或行为证据时仍使用不可变 commit permalink；引用公开 schema 时使用其 canonical `$id`，并可同时回链 publication catalog 的 source revision。保留旧 schema 文件不代表 `main` 同时支持多个 active formats。

### 4. 兼容层与迁移工具由真实需求触发

只有满足至少一项条件时，才单独设计 migration tool 或限期 compatibility layer：

- 已公开发布的数据资产需要升级；
- 外部用户或集成方存在明确迁移承诺；
- release policy 对某一格式声明了支持窗口；
- 大规模仓库内资产无法在同一变更中完成迁移。

迁移工具应优先是离线、显式的 `old -> current` 转换，不应默认扩大 runtime loader 的版本矩阵。引入兼容层必须记录支持期限、移除条件和测试责任。

### 5. Core domain 不以格式版本建模

`laneflow-core` 继续只拥有当前 domain types 和 invariant，不依赖 wire format 版本。空 `VehicleProfileRegistry` 仍可用于合法的当前空 profile package 或程序化输入，但不能作为旧格式兼容标志。

ADR 0007 的 `laneflow-data -> laneflow-core`、private DTO、Core normalization 与 loader I/O 边界继续有效。本 ADR 只替代 ADR 0006/0007 中“保留 v0.2 schema 并由 production loader 同时支持 v0.2/v0.3”的部分。

## 后果

正向后果：

- production loader、public API 和测试矩阵只随当前格式演进。
- 旧数据不会被静默升级或按错误语义加载。
- pre-1.0 breaking change 可以直接清理无外部价值的兼容负担。
- 将来出现真实迁移需求时，可以根据资产规模设计明确的离线转换，而不是提前冻结 runtime 分支。

成本与风险：

- 每次格式提升后，旧 fixture 不能再被当前 loader 直接读取。
- 开发者需要主动迁移仓库内资产，不能依赖 compatibility shim。
- 若未经确认就向外部发布数据资产，后续升级可能暴露未记录的迁移成本；release 前必须重新评估本策略。
- 接近 1.0 时必须另行冻结稳定性、deprecation、migration 和支持窗口政策。

## 替代方案

### Production loader 永久支持所有 pre-1.0 版本

读取旧数据方便，但每个版本都会增加 DTO、normalization、public dispatch 和测试责任。当前没有相应兼容承诺，因此拒绝。

### 不保留 `formatVersion`

代码更少，但旧数据可能进入当前 DTO 后产生误导性 shape/domain error，甚至因字段偶合被错误接受，因此拒绝。

### 把旧 schema 重新作为 active contract、但不保留 loader

这会让 current contract 同时出现多个“权威”格式并增加 validation 漂移，因此拒绝。ADR 0011 要求的 immutable publication artifact 是 distribution archive，不改变本拒绝结论。

### 在当前 loader 内自动迁移旧格式

调用方最省事，但会隐藏破坏性变化、扩大 production surface，并让迁移代码长期驻留。只有真实兼容需求出现后才重新评估，因此当前拒绝。

## 实施与复核

- #73：将 `laneflow-data`、schema、fixture 和 tests 收敛为单一 v0.3；删除 active v0.2 schema/example 与双版本 dispatch。
- #74 及后续 v0.3 实现只依赖当前 Core/data contract，不增加 v0.2 compatibility shim。
- #94：按同一政策将 active schema/fixture/private DTO/loader/current docs 原子替换为 v0.4；明确拒绝 v0.3，不增加 production compatibility shim。
- 接近 1.0 或首次向外部发布稳定资产前，必须创建独立 Issue 评估长期版本和迁移政策。

若后续需要同时维护多个 active data format、自动迁移旧数据或承诺支持窗口，应新增或 supersede 本 ADR，不得静默扩大 loader 行为。
